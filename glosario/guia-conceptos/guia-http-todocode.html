¿Qué es el Protocolo HTTP?
La base de toda comunicación en la web · 0:00 – 1:45

Cada vez que abres una página web, ves un video o usas una app, detrás ocurre una conversación silenciosa entre tu dispositivo y un servidor. El idioma que usan para hablar se llama HTTP.

📜
Protocolo
Un conjunto de reglas que deben seguirse para obtener un resultado o acceder a un recurso o servicio. Sin protocolo, cada computadora hablaría un idioma diferente.
🌐
HTTP
Hypertext Transfer Protocol. Protocolo de transferencia de hipertexto. Define el formato de los mensajes entre cliente y servidor.
🔒
HTTPS
HTTP + Secure. La versión cifrada. Todo lo que viaja está encriptado, por eso aparece el candado en el navegador.
💻
Cliente
El que hace la solicitud. Puede ser un navegador, una app móvil, Postman, o cualquier programa que consuma una API.
🖥
Servidor
El que responde la solicitud. Tiene los datos o recursos que el cliente necesita. Puede ser una API, un servidor web, etc.
📦
Recurso
Cualquier cosa que se puede pedir al servidor: una página HTML, una imagen, un JSON con datos de usuarios, un video, etc.

📮 Analogía: el sistema postal internacional

Imagina que quieres enviar un paquete de México a Japón. Para que llegue, necesitas seguir un protocolo: poner la dirección en el formato correcto, pagar el franqueo, usar la caja adecuada. Sin ese protocolo, el paquete no llegaría. HTTP es exactamente eso — el protocolo que hace que los mensajes entre tu dispositivo y cualquier servidor del mundo lleguen correctamente, sin importar que uno sea un iPhone en Veracruz y el otro un servidor en Tokio.

💡 ¿HTTP es solo para páginas web?

No. HTTP es el protocolo base de las APIs REST — la forma en que se comunican la mayoría de apps modernas. Cuando Instagram carga tus fotos, cuando Google Maps calcula una ruta o cuando Spotify reproduce una canción, todo eso pasa por HTTP. Es la columna vertebral del internet moderno.

El ciclo Request → Response
Cómo viajan los mensajes entre cliente y servidor · 1:45 – 3:10

HTTP funciona con un ciclo muy simple: el cliente hace una solicitud (Request) y el servidor responde con una respuesta (Response). Siempre en ese orden — nunca el servidor habla primero.

El ciclo paso a paso

El cliente (navegador, app, Postman) decide que necesita un recurso: una página, una lista de usuarios, una imagen.

El cliente construye un HTTP Request con: el método (qué quiere hacer), la URL (dónde está el recurso), y opcionalmente datos en el cuerpo.

El request viaja por internet hasta el servidor.

El servidor procesa la solicitud: busca el recurso, ejecuta la lógica necesaria, consulta la base de datos si hace falta.

El servidor construye un HTTP Response con: un código de estado (si salió bien o mal) y el cuerpo de la respuesta (los datos pedidos).

La respuesta viaja de vuelta al cliente, que la procesa y la muestra al usuario.

🍕 Analogía: pedir pizza por teléfono

Tú eres el cliente. La pizzería es el servidor. Cuando llamas a pedir pizza, estás haciendo un Request (solicitud): "Quiero una pizza hawaiana grande para llevar". La pizzería te responde (Response): "Listo, en 30 minutos" — eso es el código 200 OK. Si pides algo que no tienen: "Lo sentimos, no tenemos esa pizza" — eso sería un 404 Not Found. El teléfono es HTTP: el protocolo que permite que la conversación tenga sentido para ambas partes.

HTTP Request ejemploGET /usuarios/42 HTTP/1.1 Host: api.ejemplo.com Accept: application/json Authorization: Bearer eyJhbGci...
HTTP Response ejemploHTTP/1.1 200 OK Content-Type: application/json Date: Thu, 25 Jun 2026 10:00:00 GMT { "id": 42, "nombre": "Castillo", "email": "castillo@email.com" }

Anatomía de un Request y un Response
Qué partes tiene cada mensaje HTTP · 3:10 – 4:40

Los mensajes HTTP tienen una estructura definida. Conocerla te permite entender los errores, construir mejores APIs y usar herramientas como Postman de forma efectiva.

Partes del HTTP Request

Método (Verb)Qué acción quiere hacer el clienteGET, POST, DELETE

✉️ Analogía: el sobre de correo

Un Request HTTP es como un sobre de correo bien preparado. La dirección del destinatario es la URL. El tipo de envío (urgente, certificado, normal) es el método HTTP. Las instrucciones especiales escritas en el sobre (frágil, refrigerar, etc.) son los headers. Y la carta o paquete dentro del sobre es el body. Sin sobre correctamente escrito, el correo no sabe cómo procesarlo.

🟢 ¿Qué es una URL y qué es un Endpoint?

  • URL — La dirección completa: https://api.ejemplo.com/v1/usuarios/42
  • Endpoint — La ruta específica de un recurso en una API: /v1/usuarios/42. Es la parte de la URL que identifica qué recurso estás pidiendo.
  • API — Conjunto de endpoints disponibles en un servidor para que otros programas se comuniquen con él.

Los métodos HTTP — verbos de acción
Qué acción quieres hacer sobre el recurso · 4:40 – 5:30

El método HTTP le dice al servidor qué quieres hacer con el recurso. Son como verbos: pedir, crear, actualizar, borrar. Cada uno tiene una semántica clara y no deben usarse de forma intercambiable.

GET
READ — Leer

Obtener/leer un recurso del servidor. No modifica nada.

POST
CREATE — Crear

Enviar datos al servidor para crear un nuevo recurso.

PUT
UPDATE completo

Reemplazar un recurso existente en su totalidad.

PATCH
UPDATE parcial

Actualizar solo algunos campos de un recurso existente.

DELETE
DELETE — Eliminar

Eliminar un recurso específico del servidor.

🗂 Analogía: el archivista de una oficina

Imagina que trabajas en una oficina con un archivista. GET es decirle "muéstrame el expediente 42". POST es "crea un expediente nuevo con estos datos". PUT es "reemplaza completamente el expediente 42 por esta versión nueva". PATCH es "solo actualiza el teléfono del expediente 42". DELETE es "destruye el expediente 42". El archivista (servidor) sabe exactamente qué hacer porque le dijiste el verbo correcto.

💡 CRUD y los métodos HTTP

En programación, las 4 operaciones básicas sobre datos se llaman CRUD: Create, Read, Update, Delete. Los métodos HTTP mapean directamente a CRUD: POST → Create, GET → Read, PUT/PATCH → Update, DELETE → Delete. Toda API REST bien diseñada sigue esta correspondencia.

GET — Obtener recursos
El método más usado en la web · 5:30 – 7:00

GET es el método que el navegador usa automáticamente cuando escribes una URL. Sirve para leer o consultar datos. Es seguro: nunca modifica nada en el servidor.

Características del GET

  • Seguro: No modifica el estado del servidor — solo lee.
  • Idempotente: Hacer el mismo GET 100 veces da el mismo resultado.
  • Sin body: Los datos van en la URL (parámetros de consulta), nunca en el cuerpo.
  • Cacheado: Los navegadores guardan las respuestas GET en caché para ir más rápido.
HTTP// Obtener lista de todos los usuarios GET /api/usuarios HTTP/1.1 Host: api.ejemplo.com // Obtener usuario específico por ID GET /api/usuarios/42 HTTP/1.1 Host: api.ejemplo.com // Filtrar con parámetros de consulta GET /api/usuarios?ciudad=Veracruz&activo=true HTTP/1.1 Host: api.ejemplo.com

🔍 Analogía: buscar en el catálogo de una biblioteca

GET es como usar el catálogo de búsqueda de una biblioteca. Puedes buscar todos los libros de cierto autor, pedir ver un libro específico por su código, filtrar por género o año. Lo que no puedes hacer (ni deberías) es modificar el catálogo solo por buscarlo. Consultas, no modificas — eso es GET.

POST — Crear recursos
Enviar datos para crear algo nuevo · 7:00 – 8:30

POST se usa para crear un nuevo recurso en el servidor. Los datos van en el body del request, no en la URL. Cada POST exitoso normalmente crea un recurso diferente.

Características del POST

  • No idempotente: Si haces el mismo POST dos veces, creas dos recursos distintos.
  • Tiene body: Los datos del nuevo recurso van en el cuerpo del request (generalmente en JSON).
  • Respuesta 201: Un POST exitoso que crea un recurso devuelve el código 201 Created.
  • No es seguro: Sí modifica el servidor — crea datos nuevos.
HTTP// Crear un usuario nuevo POST /api/usuarios HTTP/1.1 Host: api.ejemplo.com Content-Type: application/json { "nombre": "Castillo", "email": "castillo@email.com", "ciudad": "Orizaba" } // Respuesta del servidor: HTTP/1.1 201 Created { "id": 101, // ← el servidor asigna el ID "nombre": "Castillo", "email": "castillo@email.com" }

📝 Analogía: rellenar un formulario de inscripción

POST es como entregar un formulario de inscripción a una escuela. Tú llevas el formulario con tus datos (body del request), lo entregas en la ventanilla (endpoint POST), y la escuela crea tu expediente (nuevo recurso en la base de datos) y te da un número de inscripción (el ID que devuelve el servidor). Si lo entregas dos veces, creas dos expedientes — por eso POST no es idempotente.

PUT — Actualización completa
Reemplazar un recurso entero · 8:30 – 10:00

PUT se usa para reemplazar completamente un recurso existente. Debes enviar todos los campos del recurso, no solo los que cambiaron. Si omites un campo, ese campo se elimina o se queda vacío.

Características del PUT

  • Idempotente: Hacer el mismo PUT varias veces deja el recurso en el mismo estado.
  • Reemplaza todo: Si no mandas un campo, se pierde o se pone en null.
  • Requiere ID: Se hace sobre un recurso específico — la URL incluye el ID.
  • Respuesta 200: Un PUT exitoso devuelve 200 OK con el recurso actualizado.
HTTP// Actualizar COMPLETAMENTE al usuario 42 PUT /api/usuarios/42 HTTP/1.1 Content-Type: application/json { "id": 42, "nombre": "Castillo Actualizado", "email": "nuevo@email.com", "ciudad": "Orizaba", "telefono": "555-1234" // ← todos los campos } // ⚠️ Si omites "telefono", en muchos servidores queda en null

⚠️ PUT ≠ PATCH — el error más común

PUT reemplaza todo el recurso. Si tienes un usuario con 10 campos y solo mandas 3 en el PUT, los otros 7 campos pueden perderse. Si solo quieres cambiar el email, usa PATCH — que solo actualiza lo que le mandas.

📄 Analogía: reemplazar una hoja de datos en un archivero

PUT es como sacar la hoja del expediente 42 del archivero, botarla a la basura, e insertar una hoja completamente nueva con todos los datos actualizados. Tienes que escribir toda la información, aunque solo haya cambiado un dato. Si dejas un campo en blanco en la hoja nueva, ese dato queda vacío — aunque antes tuviera información.

PATCH — Actualización parcial
Cambiar solo lo que necesitas · 10:00 – 11:15

PATCH actualiza parcialmente un recurso. Solo envías los campos que quieres cambiar — el resto permanece intacto. Es más eficiente que PUT cuando solo necesitas modificar uno o dos campos.

Características del PATCH

  • Solo lo que cambia: No necesitas enviar todo el objeto — solo los campos a modificar.
  • Más eficiente: Menos datos viajan por la red.
  • Idempotente: El mismo PATCH aplicado varias veces produce el mismo resultado.
  • Respuesta 200: Devuelve el recurso con los cambios aplicados.
HTTP// Solo actualizar el email del usuario 42 PATCH /api/usuarios/42 HTTP/1.1 Content-Type: application/json { "email": "nuevo-email@ejemplo.com" } // El nombre, ciudad, teléfono y demás datos NO se tocan // Comparación directa PUT vs PATCH: // PUT → "Reemplaza TODO el usuario con esto" (hoja nueva) // PATCH → "Solo cambia el email, deja lo demás igual" (corrección)

✏️ Analogía: corregir con liquid paper

PATCH es como aplicar liquid paper solo sobre el error en un documento. No rehaces la hoja completa — solo tapas el dato incorrecto y escribes el nuevo. Si en el expediente 42 solo cambió el teléfono, usas PATCH y solo corriges ese campo. Mucho más práctico que PUT cuando el cambio es pequeño.

DELETE — Eliminar recursos
Borrar un recurso del servidor · 11:15 – 12:20

DELETE elimina un recurso específico del servidor. La URL debe incluir el identificador del recurso a borrar. Normalmente no lleva body.

Características del DELETE

  • Idempotente: Si borras el mismo recurso dos veces, el efecto final es el mismo (el recurso ya no existe).
  • Sin body: El ID del recurso va en la URL, no en el cuerpo.
  • Respuesta 200 o 204: 200 OK si devuelve el recurso eliminado. 204 No Content si no devuelve nada.
  • 404 si no existe: Si intentas borrar algo que ya no existe, el servidor devuelve 404.
HTTP// Eliminar al usuario con ID 42 DELETE /api/usuarios/42 HTTP/1.1 Host: api.ejemplo.com // Respuesta exitosa (opción 1 — con confirmación) HTTP/1.1 200 OK { "mensaje": "Usuario 42 eliminado correctamente" } // Respuesta exitosa (opción 2 — sin contenido) HTTP/1.1 204 No Content // Si ya no existía: HTTP/1.1 404 Not Found

🗑 Analogía: destruir un expediente

DELETE es como decirle al archivista "tritura el expediente número 42". El archivista busca el expediente (si no existe, te dice que no lo encontró — 404). Si existe, lo tritura y te confirma que está hecho. Si lo pides de nuevo, el resultado es el mismo: el expediente ya no existe (idempotente). A diferencia del POST, hacer DELETE dos veces no crea un segundo borrado — el primero ya lo eliminó.

Códigos de estado HTTP — el lenguaje de las respuestas
Qué significan los números que devuelve el servidor · 12:20 – 13:31

El status code es el número que el servidor pone en cada Response para indicar qué pasó. Están agrupados en familias por el primer dígito:

FamiliaRangoSignificado general
✅ 2xx — Éxito200–299La solicitud se procesó correctamente
↪️ 3xx — Redirección300–399El recurso se movió, sigue otra URL
❌ 4xx — Error del cliente400–499El cliente hizo algo mal (URL, datos, permisos)
💥 5xx — Error del servidor500–599El servidor falló procesando una solicitud válida

Los más importantes

🚦 Analogía: los semáforos de la carretera

Los status codes son como las señales de tránsito en la carretera HTTP. Verde (2xx) — sigue adelante, todo en orden. Amarillo (3xx) — detente un momento, hay un desvío, ve por aquí. Rojo de tu culpa (4xx) — tú manejaste mal, ¿traes licencia? ¿vas por la ruta correcta? Rojo del camino (5xx) — la carretera está en mal estado, no es tu culpa, el problema es del servidor.

Resumen — Todo el protocolo HTTP en una sola vista
Lo que explica el video de TodoCode, consolidado

🟢 Los 5 conceptos clave del video

  • HTTP es el protocolo (conjunto de reglas) que permite la comunicación entre clientes y servidores en la web.
  • El ciclo es siempre: Request (cliente pide) → Response (servidor responde).
  • GET lee · POST crea · PUT reemplaza todo · PATCH modifica parte · DELETE elimina.
  • PUT vs PATCH: PUT necesita todos los campos, PATCH solo los que cambian.
  • Los status codes (2xx éxito, 4xx error cliente, 5xx error servidor) son el idioma del servidor para decirte qué pasó.
