Subconsultas (Subqueries) — Consultas dentro de consultas
Resolver problemas en pasos, anidando SELECT dentro de SELECT

Una subconsulta es un SELECT que vive dentro de otra consulta SQL. Se usa cuando necesitas el resultado de una consulta como entrada para otra.

🪆 Analogía: las muñecas rusas (matrioskas)

Una subconsulta es como una muñeca rusa: hay una consulta principal que contiene otra consulta más pequeña en su interior. Primero se resuelve la consulta de adentro (la muñeca pequeña), y su resultado se usa como ingrediente para resolver la consulta de afuera (la muñeca grande). Por ejemplo: "encuentra los productos que cuestan más que el precio promedio" — primero hay que calcular el promedio (subconsulta), y después comparar cada producto contra ese promedio (consulta externa).

Subconsultas en WHERE

SQLUSE cafe_curso; -- Productos más caros que el promedio SELECT nombre, precio FROM menu WHERE precio > ( SELECT AVG(precio) FROM menu -- subconsulta: calcula el promedio ); -- Clientes que SÍ han hecho pedidos (usando IN) SELECT nombre, email FROM clientes WHERE id IN ( SELECT DISTINCT id_cliente FROM pedidos ); -- Clientes que NUNCA han pedido (usando NOT IN) SELECT nombre, email FROM clientes WHERE id NOT IN ( SELECT DISTINCT id_cliente FROM pedidos ); -- El producto más caro (usando subconsulta con MAX) SELECT nombre, precio FROM menu WHERE precio = (SELECT MAX(precio) FROM menu);

Subconsultas en SELECT (columna calculada)

SQL-- Mostrar cada producto y cuánto se aleja del promedio SELECT nombre, precio, (SELECT ROUND(AVG(precio),2) FROM menu) AS 'promedio_general', precio - (SELECT AVG(precio) FROM menu) AS 'diferencia' FROM menu;

Subconsultas en FROM (tabla derivada)

SQL-- Usar el resultado de una consulta como si fuera una tabla SELECT categoria, promedio FROM ( SELECT categoria, AVG(precio) AS promedio FROM menu GROUP BY categoria ) AS resumen WHERE promedio > 65;

💡 Subconsulta vs JOIN — ¿cuándo usar cada uno?

Si necesitas combinar columnas de varias tablas para mostrarlas juntas → usa JOIN. Si solo necesitas filtrar usando un valor calculado de otra tabla (sin mostrar sus columnas) → la subconsulta suele ser más clara. En la práctica, muchas consultas con subconsultas se pueden reescribir con JOIN y viceversa — con la experiencia aprenderás cuál es más legible para cada caso.

Vistas (VIEWS) — Consultas guardadas como tablas virtuales
Reutilizar lógica compleja sin repetir código

Una vista es un SELECT guardado con un nombre. Se comporta como una tabla virtual: puedes hacer SELECT sobre ella como si fuera una tabla real, pero internamente solo es una consulta predefinida que se ejecuta cada vez que la usas.

📐 Analogía: la plantilla de documento guardada

Si cada semana tienes que armar el mismo informe combinando datos de varias hojas de cálculo, lo más sensato es crear una plantilla una sola vez y reutilizarla. Una vista es exactamente eso: en vez de escribir el mismo JOIN complejo de 5 tablas cada vez que necesitas un reporte, lo guardas una vez como vista y después solo escribes SELECT * FROM mi_vista.

SQLUSE cafe_curso; -- Crear una vista con el reporte de ventas por cliente CREATE VIEW vista_ventas_clientes AS SELECT c.nombre AS 'cliente', c.ciudad, COUNT(p.id) AS 'total_pedidos', SUM(m.precio * p.cantidad) AS 'total_gastado' FROM clientes c INNER JOIN pedidos p ON c.id = p.id_cliente INNER JOIN menu m ON p.id_producto = m.id GROUP BY c.id, c.nombre, c.ciudad; -- Usar la vista — se ve y se usa como una tabla normal SELECT * FROM vista_ventas_clientes ORDER BY total_gastado DESC; -- Filtrar sobre una vista, como en cualquier tabla SELECT * FROM vista_ventas_clientes WHERE total_gastado > 100; -- Ver todas las vistas existentes SHOW FULL TABLES WHERE table_type = 'VIEW'; -- Modificar una vista existente CREATE OR REPLACE VIEW vista_ventas_clientes AS SELECT c.nombre, SUM(m.precio * p.cantidad) AS 'total' FROM clientes c JOIN pedidos p ON c.id = p.id_cliente JOIN menu m ON p.id_producto = m.id GROUP BY c.id; -- Eliminar una vista DROP VIEW IF EXISTS vista_ventas_clientes;

🟢 Ventajas de usar vistas

  • Reutilización: escribes la lógica compleja una sola vez
  • Seguridad: puedes dar acceso a una vista sin exponer las tablas reales completas
  • Simplicidad: el resto del equipo solo necesita saber SELECT * FROM vista
  • Mantenimiento: si la lógica cambia, actualizas la vista una vez y todo lo que la usa se actualiza automáticamente

⚠️ Las vistas no almacenan datos

Una vista no guarda una copia de los datos — ejecuta la consulta original cada vez que la usas. Si los datos de las tablas base cambian, la vista refleja el cambio automáticamente en la siguiente consulta. No es una "foto" fija, es una "ventana" siempre actualizada.

Índices — Acelerar las búsquedas
CREATE INDEX para optimizar el rendimiento de consultas

Un índice es una estructura adicional que MySQL mantiene para encontrar filas más rápido, sin tener que revisar toda la tabla fila por fila.

📖 Analogía: el índice de un libro

Si buscas el capítulo "Triggers" en un libro de 500 páginas sin índice, tendrías que hojear página por página. Con el índice al final del libro, vas directo a "Triggers, página 412" y llegas en segundos. Un índice en SQL hace exactamente eso: mantiene una estructura ordenada que le dice a MySQL "el cliente con id=347 está exactamente aquí", evitando recorrer toda la tabla.

SQL-- Crear un índice en una columna que se consulta frecuentemente CREATE INDEX idx_email ON clientes(email); CREATE INDEX idx_ciudad ON clientes(ciudad); -- Índice en columnas usadas para JOIN (muy recomendado) CREATE INDEX idx_id_cliente ON pedidos(id_cliente); -- Ver los índices de una tabla SHOW INDEX FROM clientes; -- Eliminar un índice DROP INDEX idx_email ON clientes; -- Analizar qué tan rápida es una consulta (EXPLAIN) EXPLAIN SELECT * FROM clientes WHERE email = 'ana@gmail.com';

⚠️ Los índices tienen un costo

Los índices aceleran las lecturas (SELECT), pero ralentizan las escrituras (INSERT, UPDATE, DELETE) — porque MySQL tiene que actualizar el índice cada vez que cambian los datos. Las reglas prácticas: indexa columnas que usas frecuentemente en WHERE, JOIN y ORDER BY; no indexes todo, ni columnas que casi nunca consultas.

💡 PRIMARY KEY ya es un índice

Cuando defines PRIMARY KEY en una columna, MySQL crea automáticamente un índice sobre ella. Por eso buscar por id siempre es rápido sin que tengas que hacer nada extra. Lo mismo pasa con columnas UNIQUE.

Procedimientos Almacenados (Stored Procedures)
Bloques de lógica SQL reutilizables, con parámetros

Un procedimiento almacenado es un bloque de código SQL guardado en la base de datos, que se puede ejecutar (llamar) cuando lo necesites, recibiendo parámetros como una función.

🤖 Analogía: el botón de macro de la fábrica

Imagina una máquina de fábrica con un botón "ARMAR PEDIDO" que ejecuta automáticamente toda una secuencia: verifica stock, descuenta inventario, registra la venta, genera el ticket. En vez de hacer cada paso manualmente cada vez, presionas el botón con los parámetros (qué producto, cuánta cantidad) y la máquina ejecuta toda la secuencia. Un procedimiento almacenado es ese botón: empaqueta lógica compleja de varios pasos en una sola llamada.

Sintaxis básica

SQLUSE cafe_curso; -- Cambiar el delimitador (porque el procedimiento usa ; internamente) DELIMITER // CREATE PROCEDURE obtener_pedidos_cliente(IN cliente_id INT) BEGIN SELECT m.nombre AS 'producto', p.cantidad, m.precio * p.cantidad AS 'subtotal' FROM pedidos p JOIN menu m ON p.id_producto = m.id WHERE p.id_cliente = cliente_id; END // DELIMITER ; -- Llamar (ejecutar) el procedimiento CALL obtener_pedidos_cliente(1); -- pedidos de Ana García CALL obtener_pedidos_cliente(3); -- pedidos de María López

Procedimiento con parámetros de entrada y salida

SQLDELIMITER // CREATE PROCEDURE total_gastado_cliente( IN p_cliente_id INT, OUT p_total DECIMAL(10,2) ) BEGIN SELECT COALESCE(SUM(m.precio * p.cantidad), 0) INTO p_total FROM pedidos p JOIN menu m ON p.id_producto = m.id WHERE p.id_cliente = p_cliente_id; END // DELIMITER ; -- Usar el procedimiento con OUT CALL total_gastado_cliente(3, @total); SELECT @total AS 'total_gastado_maria';

Procedimiento que inserta datos (con lógica)

SQLDELIMITER // CREATE PROCEDURE registrar_pedido( IN p_cliente INT, IN p_producto INT, IN p_cantidad INT ) BEGIN -- Validar que el producto esté disponible antes de insertar IF (SELECT disponible FROM menu WHERE id = p_producto) = TRUE THEN INSERT INTO pedidos (id_cliente, id_producto, cantidad) VALUES (p_cliente, p_producto, p_cantidad); SELECT 'Pedido registrado correctamente' AS 'resultado'; ELSE SELECT 'Error: producto no disponible' AS 'resultado'; END IF; END // DELIMITER ; CALL registrar_pedido(2, 4, 2); -- Luis pide 2 Matcha Latte

🟢 ¿Por qué cambiar el DELIMITER?

El punto y coma ; normalmente le dice a MySQL "fin de la instrucción". Pero dentro de un procedimiento necesitas varios ; para separar las líneas internas sin que MySQL crea que el procedimiento ya terminó. Por eso temporalmente cambiamos el delimitador a //, escribimos todo el procedimiento, y al final regresamos el delimitador a ; con DELIMITER ;.

Triggers (Disparadores) — Acciones automáticas
Código que se ejecuta solo cuando algo sucede en la tabla

Un trigger es código que se ejecuta automáticamente cuando ocurre un evento en una tabla: antes o después de un INSERT, UPDATE o DELETE. A diferencia de los procedimientos, nunca los "llamas" tú — se disparan solos.

🚨 Analogía: la alarma de la puerta del refrigerador

El refrigerador de una cocina industrial tiene un sensor: si la puerta queda abierta más de 30 segundos, suena una alarma automáticamente. Nadie tiene que activarla manualmente — el sensor reacciona al evento (puerta abierta). Un trigger funciona igual: defines "cuando pase X en esta tabla, ejecuta automáticamente Y" — y MySQL lo dispara solo, sin que nadie tenga que llamarlo.

Trigger BEFORE INSERT — validar antes de guardar

SQLDELIMITER // CREATE TRIGGER validar_precio_positivo BEFORE INSERT ON menu FOR EACH ROW BEGIN IF NEW.precio <= 0 THEN SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'El precio debe ser mayor a cero'; END IF; END // DELIMITER ; -- Esto FALLARÁ gracias al trigger: INSERT INTO menu (nombre, precio, categoria) VALUES ('Producto gratis', -10, 'bebida'); -- Error 1644: El precio debe ser mayor a cero

Trigger AFTER INSERT — registrar en log de auditoría

SQL-- Primero, una tabla para guardar el historial CREATE TABLE IF NOT EXISTS log_pedidos ( id INT AUTO_INCREMENT PRIMARY KEY, id_pedido INT, mensaje VARCHAR(255), fecha DATETIME DEFAULT NOW() ); DELIMITER // CREATE TRIGGER log_nuevo_pedido AFTER INSERT ON pedidos FOR EACH ROW BEGIN INSERT INTO log_pedidos (id_pedido, mensaje) VALUES ( NEW.id, CONCAT('Nuevo pedido #', NEW.id, ' del cliente ', NEW.id_cliente) ); END // DELIMITER ; -- Cada vez que se inserte un pedido, se registra automáticamente en el log INSERT INTO pedidos (id_cliente, id_producto, cantidad) VALUES (5, 2, 1); SELECT * FROM log_pedidos;

Trigger AFTER UPDATE — actualizar stock automáticamente

SQLDELIMITER // CREATE TRIGGER descontar_stock_al_pedir AFTER INSERT ON pedidos FOR EACH ROW BEGIN UPDATE menu SET stock = stock - NEW.cantidad WHERE id = NEW.id_producto; END // DELIMITER ; -- Ahora cada nuevo pedido descuenta el stock automáticamente, sin código extra -- Ver todos los triggers de la base de datos SHOW TRIGGERS; -- Eliminar un trigger DROP TRIGGER IF EXISTS descontar_stock_al_pedir;

💡 NEW y OLD — las variables mágicas de los triggers

NEW hace referencia a los valores que se están insertando o al nuevo valor en un UPDATE. OLD hace referencia al valor anterior en un UPDATE o DELETE. En un INSERT solo existe NEW. En un DELETE solo existe OLD. En un UPDATE existen ambos — puedes comparar OLD.precio contra NEW.precio para saber si cambió.

⚠️ Los triggers son invisibles — úsalos con moderación

Un problema común con triggers: el código se ejecuta automáticamente y "en silencio", lo que puede confundir a otros desarrolladores que no saben que existen. Si haces un INSERT y misteriosamente otra tabla cambia, probablemente hay un trigger detrás. Documenta bien tus triggers y úsalos solo cuando la automatización realmente simplifica el sistema.
