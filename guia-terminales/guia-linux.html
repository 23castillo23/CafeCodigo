¿Qué es la terminal y por qué aprenderla?
La shell — la interfaz de texto de Linux

La terminal es una ventana donde escribes comandos de texto para darle instrucciones al sistema operativo. El programa que interpreta esos comandos se llama shell, y la más usada en Linux es Bash (Bourne Again SHell). A diferencia del CMD de Windows, en Linux la terminal no es opcional: buena parte de la administración de servidores, del desarrollo de software y de la ciencia de datos ocurre exclusivamente ahí.

🎙️ Analogía: hablarle directamente al sistema operativo

Si el escritorio con íconos es como comunicarte por señas, la terminal es como hablarle directo al oído del sistema operativo. En vez de buscar la carpeta y arrastrarla al bote de basura, le dices "borra este archivo". Un chef con cuchillo en mano trabaja distinto a uno que solo usa una picadora automática: ambos cocinan, pero uno tiene control total sobre cada corte.

¿Por qué la terminal es imprescindible en Linux?

💡 Shell, terminal y consola — no son lo mismo

La terminal es la ventana donde escribes (la aplicación gráfica). La shell es el programa que interpreta lo que escribes y lo ejecuta — Bash es la más común, pero también existen Zsh, Fish o Dash. La consola tradicionalmente se refiere a la pantalla física de acceso directo al sistema, sin entorno gráfico. En la práctica cotidiana, la gente usa los tres términos casi como sinónimos.

💡 ¿Y si vengo de CMD?

  • dir → ls — listar archivos
  • cd se comporta casi igual
  • copy/move/del → cp/mv/rm
  • Las rutas usan / en vez de \, y Linux distingue mayúsculas de minúsculas (Foto.jpg ≠ foto.jpg)
  • No hay letras de unidad (C:, D:) — todo cuelga de una sola raíz /

⚠️ Usuario normal vs superusuario (root)

Un usuario normal solo puede modificar sus propios archivos. El superusuario, o root, tiene acceso total al sistema. En vez de iniciar sesión como root todo el tiempo (peligroso), se usa sudo ("superuser do") para ejecutar un solo comando con privilegios elevados. Un error con sudo puede dañar el sistema — úsalo con cuidado y solo cuando lo necesites.

Rutas en Linux — absoluta vs relativa

🗺️ Analogía: la dirección de tu casa

Una ruta absoluta es la dirección completa: "Av. Reforma 123, Col. Centro" — funciona desde cualquier lugar. Una ruta relativa es como decirle a un vecino "está a dos cuadras al norte" — solo funciona si ambos parten del mismo punto. En Linux toda ruta absoluta empieza en la raíz /; la relativa parte de donde estés parado en ese momento.

Rutas absolutas vs relativas y atajos
-- Ruta ABSOLUTA: empieza desde la raíz / ~$ cd /home/hugo/proyectos -- Ruta RELATIVA: parte desde donde estás ahora hugo@cafe-server:~$ cd proyectos -- (mismo resultado si ya estabas en /home/hugo) -- Caracteres especiales de rutas: ~$ cd .. ← subir un nivel (directorio padre) ~$ cd ../.. ← subir dos niveles ~$ cd ~ ← ir a tu carpeta personal (home) ~$ cd / ← ir a la raíz del sistema ~$ cd - ← volver al directorio anterior ~$ cd . ← el directorio actual (se usa mucho en otros comandos)

💡 El árbol de directorios de Linux

A diferencia de Windows, Linux no tiene letras de unidad — todo el sistema cuelga de una única raíz /. Las carpetas más importantes: /home (carpetas personales de cada usuario), /etc (archivos de configuración), /bin y /usr/bin (programas ejecutables), /var (logs y datos variables), /tmp (archivos temporales que se borran al reiniciar), y /root (carpeta personal del superusuario, distinta a /).

Gestión de Archivos y Carpetas
touch · mkdir · cp · mv · rm · cat · nano
Crear archivos y carpetas
-- touch: crear un archivo vacío (o actualizar su fecha si ya existe) ~$ touch notas.txt -- mkdir: crear una carpeta ~$ mkdir proyecto -- -p = crear carpetas intermedias que no existan ~$ mkdir -p proyecto/src/componentes
Copiar, mover y eliminar
-- cp: copiar archivo ~$ cp archivo.txt ~/Backup/ -- -r = copiar carpetas completas (recursivo) ~$ cp -r proyecto/ ~/Backup/ -- mv: mover Y también sirve para renombrar ~$ mv archivo.txt ~/Documentos/ ~$ mv viejo.txt nuevo.txt ← "renombrar" es mover al mismo lugar con otro nombre -- rm: eliminar (¡permanente, no hay papelera!) ~$ rm archivo.txt -- -r = eliminar carpetas con contenido ~$ rm -r carpeta/ -- -i = pedir confirmación antes de cada borrado (recomendado) ~$ rm -ri carpeta/

⚠️ rm no tiene papelera de reciclaje

En Linux, rm borra de forma permanente e inmediata — no hay "restaurar desde papelera". El comando más peligroso que existe es rm -rf / (borrar todo desde la raíz sin preguntar); los sistemas modernos lo bloquean por defecto, pero variantes con rutas mal escritas siguen siendo un riesgo real. Piensa dos veces antes de usar -f (forzar, sin preguntar).

Ver y editar contenido
-- cat: mostrar el contenido completo de un archivo ~$ cat config.txt -- less: ver archivo largo página por página (q para salir) ~$ less log.txt -- head / tail: ver solo el inicio o el final de un archivo ~$ head -n 10 log.txt ← primeras 10 líneas ~$ tail -n 10 log.txt ← últimas 10 líneas ~$ tail -f log.txt ← seguir el archivo en vivo (muy usado con logs de servidor) -- nano: editor de texto simple dentro de la terminal ~$ nano config.txt -- Ctrl+O guarda, Ctrl+X sale -- echo: mostrar texto / crear o agregar contenido a un archivo ~$ echo "Hola mundo" Hola mundo ~$ echo "Hola" > archivo.txt ← crea/sobreescribe archivo.txt

💡 nano vs vim

nano es amigable para principiantes: los atajos aparecen en pantalla y funciona como cualquier editor de texto. vim es mucho más poderoso pero tiene una curva de aprendizaje empinada — funciona por "modos" (normal, inserción, comando) y no se cierra con Ctrl+X. Si te encuentras dentro de vim sin saber cómo salir: presiona Esc, luego escribe :q y Enter.

Permisos — Quién puede hacer qué
chmod · chown · sudo — el corazón de la seguridad en Linux

Cada archivo en Linux tiene un dueño, un grupo y tres tipos de permisos: leer (r), escribir (w) y ejecutar (x). Esto es algo que el CMD de Windows apenas expone, pero en Linux es central: define quién puede abrir, modificar o correr cada archivo.

🔑 Analogía: las llaves de un edificio de oficinas

Piensa en cada archivo como una oficina. El dueño tiene su propio juego de llaves. El grupo (por ejemplo, el equipo de contabilidad) comparte otro juego. Y "otros" es cualquier persona que entra al edificio. Cada juego de llaves puede abrir la puerta (leer), reordenar los muebles (escribir) o usar la maquinaria dentro (ejecutar) — y no todos tienen los mismos permisos.

Leer los permisos con ls -l
~$ ls -l script.sh -rwxr-xr-- 1 hugo hugo 220 jul 6 10:00 script.sh ╔═══════════════════════════════════════════════════════╗ ║ - rwx r-x r-- ← tipo + 3 grupos de permisos ║ ║ ↑ ↑ ↑ ↑ ║ ║ tipo dueño grupo otros ║ ╚═══════════════════════════════════════════════════════╝ -- tipo: - archivo normal, d carpeta, l enlace simbólico -- rwx = leer + escribir + ejecutar | r-x = leer + ejecutar | r-- = solo leer
Valor numéricoPermisosSignificado
7rwxLeer + escribir + ejecutar (4+2+1)
6rw-Leer + escribir (4+2)

0---Sin ningún permiso
chmod — cambiar permisos
-- Modo numérico: dueño-grupo-otros ~$ chmod 755 script.sh ← dueño: rwx, grupo: r-x, otros: r-x ~$ chmod 644 notas.txt ← dueño: rw-, grupo: r--, otros: r-- (típico para archivos normales) -- Modo simbólico: sumar o quitar permisos puntuales ~$ chmod +x script.sh ← dar permiso de ejecución a todos ~$ chmod u+x script.sh ← solo al dueño (u=user) ~$ chmod g-w archivo.txt ← quitar escritura al grupo (g=group)
chown y sudo
-- chown: cambiar el dueño de un archivo (requiere sudo) ~$ sudo chown hugo archivo.txt ~$ sudo chown hugo:desarrollo archivo.txt ← dueño y grupo a la vez -- sudo: ejecutar un comando puntual como superusuario ~$ sudo apt update [sudo] password for hugo: -- sudo -i: abrir una sesión completa como root (usar con mucho cuidado)

💡 ¿Por qué mis scripts .sh no se ejecutan?

Un script recién creado normalmente no tiene permiso de ejecución. Si al correr ./script.sh ves "Permission denied", la solución casi siempre es chmod +x script.sh.

locate · which
-- locate: búsqueda instantánea usando un índice pre-generado (más rápido que find) ~$ locate archivo.txt -- si el índice está desactualizado: sudo updatedb -- which: saber en qué ruta está instalado un comando ~$ which python3 /usr/bin/python3

💡 find vs grep vs locate

find busca archivos por nombre, tipo, tamaño o fecha. grep busca texto dentro del contenido de los archivos. locate también busca archivos por nombre, pero usa un índice pre-armado en vez de recorrer el disco en tiempo real — es mucho más rápido pero puede estar desactualizado.

Pipes y Redirección
Encadenar comandos y guardar su salida

🏭 Analogía: la línea de ensamblaje

Cada comando produce una "salida" y puede tomar una "entrada". El operador | (pipe) es como una banda transportadora que conecta la salida de una máquina directamente con la entrada de la siguiente, sin pasos manuales de por medio. En vez de guardar resultados intermedios, encadenas comandos hasta obtener justo lo que necesitas.

Comandos de Red
ip · ping · traceroute · ss · curl · dig

🔍 Analogía: el diagnóstico del mecánico

Los comandos de red son el escáner de diagnóstico de tu conexión: te dicen si hay señal, por dónde viajan los datos y qué está conectado en cada puerto. Igual que un mecánico revisa cable por cable antes de decir "es la batería", tú revisas capa por capa antes de culpar al Wi-Fi.

ip — tu configuración de red
-- Ver direcciones IP de todas las interfaces (reemplaza al viejo ifconfig) ~$ ip a 2: eth0: <BROADCAST,MULTICAST,UP> ... inet 192.168.1.105/24 brd 192.168.1.255 -- Ver la tabla de rutas (puerta de enlace) ~$ ip route default via 192.168.1.1 dev eth0
ping — probar conectividad
-- Ping envía paquetes hasta que lo detengas (Ctrl+C), a diferencia del CMD ~$ ping google.com 64 bytes from 142.250.65.46: icmp_seq=1 ttl=118 time=18.2 ms 64 bytes from 142.250.65.46: icmp_seq=2 ttl=118 time=17.9 ms ^C --- google.com ping statistics --- 2 packets transmitted, 2 received, 0% packet loss -- -c = número específico de paquetes (como -n en Windows) ~$ ping -c 4 8.8.8.8 -- probar la conexión local (loopback) ~$ ping -c 2 127.0.0.1
traceroute · ss · dig · curl
-- traceroute: ver el camino que hacen los datos hasta su destino ~$ traceroute google.com 1 192.168.1.1 1.2 ms ← tu router 2 10.0.0.1 8.1 ms ← tu ISP ...hasta llegar a destino -- ss: ver conexiones y puertos activos (reemplaza al viejo netstat) ~$ ss -tulnp Netid State Local Address:Port Process tcp LISTEN 0.0.0.0:80 nginx -- dig: resolver nombres de dominio a IPs (más detallado que nslookup) ~$ dig google.com ;; ANSWER SECTION: google.com. 300 IN A 142.250.65.46 -- curl: descargar contenido de una URL o probar una API ~$ curl https://api.github.com ~$ curl -O https://ejemplo.com/archivo.zip ← descargar y guardar el archivo

Gestión de Procesos
ps · top · kill · jobs · bg / fg
ps · top · kill
-- ps: ver procesos en ejecución (foto fija) ~$ ps aux USER PID %CPU %MEM COMMAND hugo 4532 2.1 3.4 /usr/bin/chrome hugo 7821 0.5 1.2 /usr/bin/code -- filtrar por nombre de proceso ~$ ps aux | grep chrome -- top: monitor de procesos en tiempo real (q para salir) ~$ top -- htop es una versión más visual (puede requerir: sudo apt install htop) -- kill: terminar un proceso por su PID ~$ kill 4532 -- kill -9 = forzar el cierre (para procesos que no responden) ~$ kill -9 4532 -- killall: terminar por nombre en vez de PID ~$ killall chrome
Procesos en segundo plano
-- & al final ejecuta el comando en segundo plano (sigues usando la terminal) ~$ npm run dev & [1] 8842 -- jobs: ver qué procesos tienes corriendo en esta sesión ~$ jobs [1]+ Running npm run dev & -- fg: traer un proceso al primer plano ~$ fg %1 -- Ctrl+Z pausa el proceso en primer plano; bg lo reanuda en segundo plano ~$ bg %1

💡 PID — el número de identidad de cada proceso

Cada programa en ejecución recibe un PID (Process ID) único mientras corre. Comandos como kill lo usan para saber exactamente a qué proceso apuntar, ya que puede haber varios procesos con el mismo nombre corriendo a la vez.

Gestión de Paquetes
apt · dpkg — instalar y actualizar software

En Linux no se descargan instaladores .exe de internet: se usa un gestor de paquetes que descarga, instala y actualiza software desde repositorios oficiales. En Ubuntu/Debian el gestor es apt; otras distros usan dnf/yum (Fedora/RHEL) o pacman (Arch).

apt — Ubuntu / Debian
-- Actualizar la lista de paquetes disponibles (siempre primero) ~$ sudo apt update -- Actualizar todos los paquetes instalados a su última versión ~$ sudo apt upgrade -- Instalar un programa ~$ sudo apt install htop -- Eliminar un programa ~$ sudo apt remove htop -- Buscar un paquete por nombre ~$ apt search nginx

💡 apt vs apt-get vs dpkg

  • apt — el comando moderno y recomendado para uso diario
  • apt-get — la versión más antigua, todavía usada en scripts
  • dpkg — el gestor de bajo nivel que instala archivos .deb individuales, por ejemplo sudo dpkg -i paquete.deb

💡 ¿Qué es el PATH?

El PATH es una lista de carpetas donde Bash busca el programa que escribiste, en orden. Por eso puedes escribir ls desde cualquier lugar sin dar la ruta completa: /bin/ls ya está en una carpeta listada en el PATH. Si instalas un programa y "no se reconoce como comando", casi siempre es porque su carpeta no está en el PATH.

💡 La línea shebang #!/bin/bash

La primera línea de todo script le dice al sistema qué intérprete usar para ejecutarlo. #!/bin/bash significa "ejecuta este archivo con Bash". Sin esta línea, el script podría interpretarse con el shell equivocado.

Condicionales y bucles
-- Condicional if #!/bin/bash if [ -f "archivo.txt" ]; then echo "El archivo existe" else echo "No se encontró" fi -- Bucle for for archivo in *.txt; do echo "Procesando $archivo" done
✏️ Ejercicio — Script de respaldo automático Avanzado
Objetivo: Escribe un script que cree una carpeta con la fecha del día y copie ahí tus archivos .txt.
backup.sh
#!/bin/bash # Guarda la fecha de hoy en una variable fecha=$(date +%Y-%m-%d) destino=~/Backup/$fecha # Crea la carpeta de destino si no existe mkdir -p "$destino" # Copia todos los .txt a esa carpeta cp *.txt "$destino"/ echo "Respaldo del $fecha completado en $destino"
