¿Qué es Git y por qué usarlo?
Control de versiones — el historial de tu proyecto

Git es un sistema de control de versiones: guarda "fotografías" del estado de tu proyecto en el tiempo, para que puedas volver a cualquier punto anterior, comparar cambios, o trabajar en varias versiones del mismo código a la vez sin pisarte a ti mismo. GitHub es distinto: es una plataforma en la nube donde alojas tus repositorios de Git y colaboras con otras personas. Git es la herramienta, GitHub es donde la usas en equipo.

🎮 Analogía: los puntos de guardado de un videojuego

Sin Git, programar es como jugar un videojuego difícil sin puntos de guardado: si algo sale mal, no hay forma de volver atrás. Git es guardar la partida cada vez que logras algo importante. Puedes volver a cualquier punto de guardado (commit), probar una estrategia arriesgada en un archivo de guardado aparte (rama) sin arruinar tu partida principal, y si la estrategia funciona, fusionas ese progreso de vuelta.

¿Por qué no basta con "Deshacer" o copiar carpetas?

🕰️
Historial real
Cada commit queda guardado para siempre, con quién lo hizo, cuándo y por qué — no solo los últimos pasos como Ctrl+Z.
🌿
Ramas paralelas
Puedes desarrollar una función nueva sin tocar el código que ya funciona, y decidir después si la integras.
🤝
Colaboración real
Varias personas editan el mismo proyecto sin sobrescribirse — Git sabe fusionar los cambios de todos.
☁️
Respaldo en la nube
Con GitHub, tu código vive también fuera de tu computadora — si se daña tu disco, el proyecto sigue ahí.
🔍
Trazabilidad
Puedes ver exactamente qué línea cambió, cuándo y por qué — clave para encontrar en qué commit se rompió algo.
💼
Estándar de la industria
Prácticamente todo trabajo de programación profesional usa Git — es una habilidad no negociable.

💡 Git ≠ GitHub

Git funciona 100% en tu computadora, sin internet — es el motor de control de versiones. GitHub (y alternativas como GitLab o Bitbucket) es un servicio en línea que aloja copias de tus repositorios Git y agrega herramientas de colaboración: Pull Requests, Issues, revisiones de código. Podrías usar Git toda tu vida sin tocar GitHub — pero en equipo, GitHub es donde ese historial se comparte.

Instalación y configuración inicial
Antes de tu primer commit
Verificar instalación
-- Comprobar si ya tienes Git instalado ~$ git --version git version 2.45.0 -- Si no aparece, descárgalo desde git-scm.com (Windows/Mac) -- En Linux (Ubuntu/Debian): ~$ sudo apt install git
Configuración inicial (una sola vez por computadora)
-- Tu identidad en cada commit (aparece en el historial) ~$ git config --global user.name "Hugo" ~$ git config --global user.email "hugo@ejemplo.com" -- Ver toda tu configuración actual ~$ git config --list -- Definir tu editor de texto por defecto para mensajes largos ~$ git config --global core.editor "nano"

💡 --global vs configuración local

--global aplica la configuración a todos tus repositorios en esa computadora. Si quieres usar un nombre o correo distinto para un proyecto específico (por ejemplo, uno de trabajo), ejecuta el mismo comando sin --global estando dentro de esa carpeta.

Conceptos fundamentales
El vocabulario que necesitas antes de escribir comandos

📸 Analogía: las tres áreas como un estudio fotográfico

El directorio de trabajo es el escenario donde mueves los objetos (editas archivos). El área de preparación (staging) es cuando acomodas y encuadras justo lo que quieres que salga en la foto. El commit es el clic del obturador: guarda esa foto para siempre en el álbum (el repositorio).

📁
Repositorio (repo)
La carpeta de tu proyecto una vez que Git la está vigilando — contiene todo el historial en una carpeta oculta .git.
🗂️
Directorio de trabajo
Los archivos tal como los ves y editas normalmente en tu carpeta.
🎯
Área de preparación (staging)
Una zona intermedia donde eliges exactamente qué cambios entrarán en el próximo commit.
📸
Commit
Una fotografía guardada del proyecto en ese momento, con un mensaje que explica qué cambió.
🌿
Rama (branch)
Una línea de desarrollo independiente — la línea principal suele llamarse main.
📍
HEAD
Un puntero que indica en qué commit / rama estás parado ahora mismo.
☁️
Remoto (remote)
Una copia del repositorio alojada en otro lugar — normalmente GitHub. El nombre por defecto es origin.
🔀
Merge
Fusionar los cambios de una rama dentro de otra.
⚡
Conflicto
Cuando dos cambios tocan la misma línea y Git no sabe cuál conservar — te pide que decidas tú.
🔑
Hash de commit
Un identificador único (ej. a3f9c2e) que Git le da a cada commit para referenciarlo.

Comandos básicos
init · status · add · commit · log
Iniciar un repositorio
-- Convertir la carpeta actual en un repositorio de Git ~/mi-proyecto$ git init Initialized empty Git repository in /home/hugo/mi-proyecto/.git/
git status — el comando que más vas a usar
~/mi-proyecto$ git status On branch main No commits yet Untracked files: (use "git add <file>..." to include in what will be committed) index.html style.css
git add — preparar cambios
-- Agregar un archivo específico al área de preparación ~$ git add index.html -- Agregar varios archivos a la vez ~$ git add index.html style.css -- Agregar TODOS los archivos modificados/nuevos ~$ git add . -- Quitar un archivo del área de preparación (sin perder el cambio) ~$ git restore --staged archivo.txt
git commit — guardar el punto de control
-- Guardar los cambios preparados con un mensaje descriptivo ~$ git commit -m "Agregar estructura inicial de la página" [main (root-commit) a3f9c2e] Agregar estructura inicial de la página 2 files changed, 40 insertions(+) -- Atajo: agregar TODOS los archivos ya rastreados y hacer commit en un paso -- (no incluye archivos completamente nuevos, solo modificados) ~$ git commit -am "Corregir bug del formulario"

⚠️ Escribe mensajes de commit que digan algo

"cambios", "arreglos", "asdasd" no ayudan a nadie — ni siquiera a ti mismo en 3 meses. Usa mensajes en modo imperativo y específicos: "Agregar validación al formulario de contacto" en vez de "cambios".

git log — ver el historial
~$ git log commit a3f9c2e1b8d4f... (HEAD -> main) Author: Hugo <hugo@ejemplo.com> Date: Thu Jul 16 10:00:00 2026 Agregar estructura inicial de la página -- Versión compacta, una línea por commit ~$ git log --oneline a3f9c2e Agregar estructura inicial de la página -- Ver también el árbol de ramas de forma visual ~$ git log --oneline --graph --all -- Ver qué cambió exactamente en los archivos (diferencias línea por línea) ~$ git diff - <h1>Bienvenido</h1> + <h1>Bienvenido a Café </></h1>

🌿
Ramas — trabajar en paralelo sin miedo
branch · checkout · switch · merge

🌌 Analogía: universos paralelos

Una rama es una línea temporal alterna de tu proyecto. Puedes crear una rama para probar una función nueva y arriesgada — si algo sale mal, simplemente abandonas esa rama y tu línea principal (main) nunca se enteró de que algo falló. Si funciona, fusionas ("merge") esa línea temporal de vuelta a la principal.

Crear y moverte entre ramas
-- Ver en qué rama estás y qué otras existen ~$ git branch * main -- Crear una rama nueva (sin moverte a ella todavía) ~$ git branch nueva-funcion -- Moverte a una rama existente ~$ git checkout nueva-funcion -- forma moderna equivalente: ~$ git switch nueva-funcion -- Crear Y moverte a la rama en un solo paso ~$ git checkout -b nueva-funcion -- forma moderna equivalente: ~$ git switch -c nueva-funcion -- Renombrar la rama actual ~$ git branch -m nombre-nuevo -- Eliminar una rama que ya fusionaste ~$ git branch -d nueva-funcion -- forzar eliminación aunque no esté fusionada (¡se pierde el trabajo!) ~$ git branch -D nueva-funcion
Fusionar ramas (merge)
-- 1. Ve a la rama que RECIBIRÁ los cambios (normalmente main) ~$ git switch main -- 2. Fusiona la otra rama dentro de main ~$ git merge nueva-funcion Updating a3f9c2e..d8b1e40 Fast-forward index.html | 12 ++++++++++++

💡 ¿Por qué "main" y no "master"?

Históricamente la rama principal se llamaba master. Desde 2020, GitHub y la mayoría de proyectos nuevos usan main como nombre por defecto. Ambos funcionan igual — es solo una convención de nombre, no una diferencia técnica.

✏️ Ejercicio — Crea y fusiona tu primera rama Intermedio
Objetivo: Practica el flujo completo: crear rama, hacer cambios, fusionar.
Flujo de rama
-- 1. Crea y muévete a una rama nueva ~$ git switch -c agregar-footer -- 2. Haz un cambio y confírmalo ~$ echo "<footer>Café </></footer>" >> index.html ~$ git commit -am "Agregar footer" -- 3. Regresa a main y fusiona ~$ git switch main ~$ git merge agregar-footer -- 4. Elimina la rama ya fusionada ~$ git branch -d agregar-footer

Deshacer cambios
restore · reset · revert — el botón de "arrepentirse"
Quiero...Comando¿Qué hace?
Descartar cambios sin confirmar en un archivogit restore archivo.txtVuelve el archivo a como estaba en el último commit
Sacar un archivo del área de preparacióngit restore --staged archivo.txtEl cambio sigue ahí, solo deja de estar "listo para commit"
Deshacer el último commit, conservando los cambiosgit reset --soft HEAD~1El commit desaparece, pero los archivos quedan preparados
Deshacer el último commit y los cambios en staginggit reset HEAD~1El commit desaparece, cambios vuelven al directorio de trabajo
Borrar el último commit por completogit reset --hard HEAD~1⚠️ Elimina el commit Y todos sus cambios, sin posibilidad de recuperarlos fácil
Deshacer un commit ya compartido con otrosgit revert <hash>Crea un commit NUEVO que anula los cambios del anterior — no borra historial

⚠️ reset --hard vs revert

git reset --hard reescribe el historial — perfecto para deshacer cosas que solo tú has visto, nunca subidas a un remoto compartido. Si el commit ya lo vieron tus compañeros de equipo (ya hiciste push), usa git revert en su lugar: crea un commit nuevo que cancela al anterior, sin borrar el historial que otros ya descargaron.

Ejemplos
-- Revertir un commit específico de forma segura ~$ git revert a3f9c2e -- Ver el historial para encontrar el hash que necesitas ~$ git log --oneline
☁️
GitHub y repositorios remotos
clone · remote · push · pull · fetch

☁️ Analogía: tu proyecto en la nube

Hasta ahora todo el historial vive solo en tu computadora. Un remoto es una copia de ese repositorio guardada en otro lugar — casi siempre GitHub. push es subir tus fotos nuevas al álbum compartido; pull es bajar las fotos que otros agregaron desde que revisaste por última vez.

Empezar desde cero en GitHub
-- 1. Crea el repositorio vacío en github.com (botón "New repository") -- 2. Conecta tu repositorio local con ese remoto ~$ git remote add origin https://github.com/hugo/mi-proyecto.git -- 3. Verifica que quedó conectado ~$ git remote -v origin https://github.com/hugo/mi-proyecto.git (fetch) origin https://github.com/hugo/mi-proyecto.git (push) -- 4. Sube tu rama principal por primera vez ~$ git push -u origin main -- -u guarda el vínculo: después de esto, basta con "git push"

⚠️ No marques README, .gitignore ni licencia al crear el repo

Si ya tienes un proyecto local y vas a subirlo con git push, crea el repositorio en GitHub completamente vacío — sin marcar "Add a README", "Add .gitignore" ni "Choose a license". Si lo dejas marcado, GitHub crea archivos que tu repositorio local no tiene, y el primer push falla porque las historias no coinciden.

🔐 La primera vez te va a pedir iniciar sesión

Al ejecutar tu primer git push, se abrirá una ventana del navegador pidiéndote autorizar el acceso a tu cuenta de GitHub — haz clic en "Sign in with your browser", confirma, y listo. Git Credential Manager guarda esa sesión, así que en los próximos push no te lo volverá a pedir.

Empezar clonando un repositorio existente
-- Descargar una copia completa (con todo el historial) a tu computadora ~$ git clone https://github.com/hugo/mi-proyecto.git Cloning into 'mi-proyecto'... -- Ya queda todo conectado: no necesitas "git remote add"
El ciclo diario: push y pull
-- Subir tus commits locales al remoto ~$ git push -- Bajar los commits nuevos del remoto y fusionarlos automáticamente ~$ git pull -- fetch: descarga los cambios del remoto SIN fusionarlos todavía -- (útil para revisar antes de mezclar) ~$ git fetch

💡 pull = fetch + merge

git pull en realidad hace dos cosas en un solo paso: descarga los cambios nuevos (fetch) y los fusiona en tu rama actual (merge). Si prefieres revisar los cambios antes de mezclarlos, usa fetch solo y decide después.

✏️ Ejercicio — Sube tu primer proyecto a GitHub Intermedio
Objetivo: Conecta un repositorio local con uno recién creado en GitHub y sube tu primer commit.
De local a GitHub
-- 1. Crea un repo vacío en GitHub (sin README) y copia su URL -- 2. Conéctalo a tu proyecto local ~$ git remote add origin URL_DE_TU_REPO -- 3. Sube tu rama principal ~$ git push -u origin main -- 4. Refresca la página de GitHub — ¡ahí está tu código!
🌍
GitHub Pages — publica tu sitio gratis
Un link público para cualquier proyecto de HTML/CSS/JS, sin pagar hosting

🪟 Analogía: el escaparate de tu tienda

Hasta ahora tu repositorio es como una bodega — guarda tu código, pero nadie de afuera lo ve funcionando. GitHub Pages convierte esa misma bodega en un escaparate: toma los archivos de tu repositorio y los publica como un sitio web real, con su propia dirección pública, gratis y sin que tengas que contratar hosting.

Activar GitHub Pages
-- 1. Entra a tu repositorio en github.com -- 2. Ve a Settings (la rueda de configuración, arriba a la derecha) -- 3. En el menú izquierdo, busca "Pages" -- 4. En la sección "Branch", cambia "None" por "main" -- 5. Deja la carpeta como / (root) -- 6. Haz clic en "Save" -- 7. Espera 1-2 minutos y recarga la página ✅ Your site is live at https://tu-usuario.github.io/tu-repositorio/

Ese link funciona en cualquier navegador, en cualquier dispositivo, sin que la otra persona instale nada. Cada vez que hagas git push a la rama que configuraste, el sitio se actualiza solo en uno o dos minutos.

⚠️ Cosas a revisar si tu sitio usa Firebase o rutas locales

Si tu proyecto usa Firebase (como Café </>), las funciones que dependen de Firebase van a seguir funcionando en GitHub Pages, porque Firebase es un servicio externo — no depende de dónde esté alojado tu HTML. Lo que sí puede fallar son rutas de archivos escritas en mayúsculas/minúsculas distintas a como se llaman de verdad: tu computadora (Windows) no distingue mayúsculas en nombres de archivo, pero el servidor de GitHub Pages corre sobre Linux y sí las distingue. Si algo no carga después de publicar, revisa primero eso.

💡 ¿Qué carpeta debería tener tu index.html?

GitHub Pages busca automáticamente un archivo index.html en la raíz de la rama que configuraste (o en la carpeta /docs si eliges esa opción). Asegúrate de que tu página principal se llame exactamente index.html y esté en el lugar correcto, o la URL pública mostrará un error 404.

Colaboración
Fork · Pull Request · conflictos de merge
🍴
Fork
Tu propia copia completa de un repositorio ajeno en tu cuenta de GitHub — para proponer cambios sin tener permiso directo sobre el original.
🔃
Pull Request (PR)
Una propuesta formal: "aquí están mis cambios, revísalos e intégralos si te parecen bien" — el corazón de la colaboración en GitHub.
👀
Code Review
El proceso de que alguien más revise tus cambios antes de fusionarlos, dejando comentarios en líneas específicas.
📌
Issue
Un reporte de error o propuesta de función, registrado en GitHub para darle seguimiento.
Flujo típico de una Pull Request
-- 1. Crea una rama para tu cambio (nunca trabajes directo en main) ~$ git switch -c corregir-boton-login -- 2. Haz tus cambios y confírmalos ~$ git commit -am "Corregir estilo del botón de login" -- 3. Sube la rama al remoto (todavía no a main) ~$ git push -u origin corregir-boton-login -- 4. Ve a GitHub → aparecerá un botón "Compare & pull request" -- ábrelo, describe tu cambio, y espera revisión/aprobación -- 5. Una vez aprobado, se fusiona desde la interfaz de GitHub
Resolver un conflicto de merge
~$ git merge otra-rama CONFLICT (content): Merge conflict in index.html Automatic merge failed; fix conflicts and then commit the result. -- Git marca el archivo así, directamente dentro del código: <<<<<<< HEAD <h1>Bienvenido a Café</h1> ======= <h1>Bienvenido a Café </></h1> >>>>>>> otra-rama -- 1. Edita el archivo a mano: decide qué línea(s) conservar -- 2. Borra las marcas <<<<<<<, ======= y >>>>>>> -- 3. Marca el conflicto como resuelto ~$ git add index.html -- 4. Completa el merge con un commit ~$ git commit -m "Resolver conflicto de merge en index.html"

✍️ Analogía: dos personas editando el mismo párrafo

Un conflicto es como si tú y un compañero editaran a mano la misma línea de un documento impreso al mismo tiempo, cada uno con su versión. Git no adivina cuál preferir — te muestra ambas versiones lado a lado y te pide que decidas tú cuál (o una combinación de las dos) es la correcta.

.gitignore y buenas prácticas
Qué NO debe entrar nunca a tu repositorio

El archivo .gitignore le dice a Git qué archivos y carpetas ignorar por completo — nunca los sube, nunca los rastrea. Es esencial para no subir archivos pesados, generados automáticamente, o con información sensible.

.gitignore de ejemplo
# Dependencias (se reinstalan con npm install, no hace falta subirlas) node_modules/ # Variables de entorno y contraseñas — ¡NUNCA subir! .env # Archivos del sistema operativo .DS_Store Thumbs.db # Carpetas de compilación / build dist/ build/ # Logs *.log

⚠️ Nunca subas contraseñas ni llaves de API

Una vez que algo entra a un commit, queda en el historial para siempre — aunque lo borres después, sigue existiendo en commits anteriores. Si accidentalmente subiste una contraseña o llave de API, no basta con borrarla en el siguiente commit: hay que revocarla/cambiarla de inmediato, porque ya es pública.

Si ya rastreabas un archivo antes de ignorarlo
-- Agregar la regla a .gitignore no basta si el archivo YA estaba rastreado -- Hay que decirle a Git que deje de rastrearlo (sin borrarlo del disco) ~$ git rm --cached archivo-secreto.env ~$ git commit -m "Dejar de rastrear archivo-secreto.env"

💡 Buenas prácticas rápidas

  • Commits pequeños y frecuentes, cada uno con un propósito claro
  • Nunca trabajes directo en main en un proyecto en equipo — usa ramas
  • Haz git pull antes de empezar a trabajar cada día
  • Revisa git status antes de cada add/commit

Avanzado
stash · rebase · cherry-pick · tags
git stash — guardar cambios sin hacer commit
-- Guardar cambios a medio hacer para cambiar de rama sin perderlos ~$ git stash Saved working directory and index state WIP on main -- Ver la lista de cambios guardados ~$ git stash list stash@{0}: WIP on main: a3f9c2e Agregar footer -- Recuperar el último cambio guardado ~$ git stash pop

🗄️ Analogía del cajón temporal

git stash es como meter todo lo que tienes a medio hacer en tu escritorio dentro de un cajón, para poder atender algo urgente en una superficie limpia. Cuando terminas lo urgente, abres el cajón y todo sigue exactamente donde lo dejaste.

git rebase — reescribir el historial de forma ordenada
-- Alternativa a merge: mueve tus commits para que queden "encima" de main -- resultando en un historial lineal, sin commits de "merge" ~$ git switch mi-rama ~$ git rebase main

⚠️ Regla de oro del rebase

Nunca hagas rebase sobre commits que ya subiste y que otras personas ya descargaron — reescribe el historial, y eso puede causarles problemas serios a tus compañeros. Úsalo solo en tus ramas locales, antes de compartirlas.

cherry-pick y tags
-- cherry-pick: traer UN commit específico de otra rama, sin fusionar todo ~$ git cherry-pick a3f9c2e -- tags: marcar un commit como un punto importante (ej. una versión) ~$ git tag v1.0.0 ~$ git push origin v1.0.0 -- Ver todos los tags del repositorio ~$ git tag

Flujo de trabajo diario
Cómo se ve todo esto en la práctica, día a día
Un día típico de trabajo con Git
-- 1. Empiezas el día trayendo lo último de main ~$ git switch main ~$ git pull -- 2. Creas una rama para la tarea de hoy ~$ git switch -c agregar-carrito-compras -- 3. Trabajas, haciendo commits pequeños y frecuentes ~$ git add . ~$ git commit -m "Agregar botón de añadir al carrito" ~$ git commit -am "Conectar carrito con el total" -- 4. Subes tu rama y abres una Pull Request en GitHub ~$ git push -u origin agregar-carrito-compras -- 5. Una vez aprobada y fusionada, limpias tu rama local ~$ git switch main ~$ git pull ~$ git branch -d agregar-carrito-compras

💡 Un patrón simple: "rama por tarea"

No necesitas un flujo complicado para empezar. La regla más útil: una rama por tarea o función, nunca trabajar directo en main, y fusionar solo cuando esa tarea esté lista y probada. Con eso ya evitas el 90% de los problemas comunes de colaboración.
