¿Qué es el Diseño Responsive?
Una página, todos los dispositivos — sin sacrificar experiencia

El Diseño Responsive (Responsive Web Design, RWD) es la técnica de construir páginas web que se adaptan automáticamente a cualquier tamaño de pantalla: celular, tablet, laptop, monitor 4K. En vez de hacer una versión diferente para cada dispositivo, haces una sola página que "fluye" y se reorganiza según el espacio disponible.

💧 Analogía: el agua y el recipiente

El agua no tiene forma propia: toma la forma del recipiente donde la viertes. En un vaso es cilíndrica, en un tazón es ancha, en una botella es larga. Una página responsive es igual: el mismo contenido toma la forma del dispositivo donde se muestra. En celular es una columna, en tablet son dos, en desktop son tres o cuatro.

Los tres pilares del Responsive Design

1️⃣ Grillas fluidas

Usar porcentajes y unidades relativas en vez de píxeles fijos. Flexbox y Grid hacen esto automáticamente.

2️⃣ Imágenes flexibles

Las imágenes que se adaptan al ancho de su contenedor. max-width: 100% es el punto de partida.

3️⃣ Media Queries

Reglas CSS que se activan solo cuando el dispositivo cumple cierta condición: ancho mínimo, orientación, etc.

La meta viewport — El punto de partida obligatorio
Sin esto, tu página en móvil se ve como si la vieras con zoom muy alejado

Antes de cualquier media query, necesitas la etiqueta meta viewport en el <head>. Sin ella, los navegadores móviles simulan una pantalla de escritorio ancha (~980px) y luego la encogen, haciendo que todo se vea diminuto.

HTML<!-- Esta línea es OBLIGATORIA para que el responsive funcione --> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <!-- width=device-width: usa el ancho real del dispositivo --> <!-- initial-scale=1.0: sin zoom inicial -->

❌ SIN viewport meta

El móvil simula 980px de ancho y lo escala hacia abajo. Todo se ve pequeñísimo, el usuario tiene que hacer zoom manual.

✅ CON viewport meta

El móvil usa su ancho real (ej: 390px). Las media queries funcionan correctamente y el texto es legible.

⚠️ user-scalable=no — Nunca lo uses

Algunos tutoriales viejos incluyen user-scalable=no o maximum-scale=1 en el viewport. Esto deshabilita el zoom del usuario, lo que es una violación de accesibilidad y está prohibido en muchas pautas de diseño. Nunca lo hagas.

Imágenes responsive — La base
max-width, object-fit y las propiedades esenciales

El primer paso para imágenes responsive es simple: que no se salgan de su contenedor. La regla de oro es una sola línea de CSS que deberías tener en todos tus proyectos.

CSS/* La regla de oro — va en tu reset/base de CSS */ img, video, iframe { max-width: 100%; /* Nunca más ancha que su contenedor */ height: auto; /* El alto se ajusta proporcionalmente */ display: block; /* Elimina el espacio fantasma debajo de imágenes */ } /* object-fit — cómo la imagen llena su contenedor */ .imagen-hero { width: 100%; height: 400px; object-fit: cover; /* Rellena sin deformar, recorta si es necesario */ object-position: center top; /* Desde dónde recortar */ } /* Valores de object-fit */ .img { object-fit: fill; } /* Deforma para llenar (por defecto) */ .img { object-fit: contain; } /* Cabe completa, deja espacio en bordes */ .img { object-fit: cover; } /* Rellena y recorta — el más usado */ .img { object-fit: none; } /* Tamaño original, se recorta */ .img { object-fit: scale-down; } /* El menor entre none y contain */

srcset — Imágenes a la medida del dispositivo
Servir la imagen correcta según la resolución de la pantalla

El atributo srcset permite ofrecer varias versiones de la misma imagen a diferentes resoluciones. El navegador elige automáticamente la más adecuada: una imagen de 400px para móvil, una de 800px para tablet, una de 1600px para desktop. Así evitas que un celular descargue una imagen enorme innecesariamente.

🍕 Analogía: el menú de tamaños de pizza

Una pizzería ofrece la misma pizza en tres tamaños: individual, mediana y familiar. No le das la familiar a quien comió solo, ni la individual a una mesa de 8 personas. El navegador hace lo mismo con srcset: le da a cada dispositivo el "tamaño" de imagen que necesita, sin desperdiciar datos.

Descriptores de ancho (w) — el más común

<img
  src="foto-800.jpg"Fallback si srcset no funciona
  srcset="foto-400.jpg 400w,imagen ancho real en px
          foto-800.jpg 800w,
          foto-1600.jpg 1600w"
  sizes="(max-width: 600px) 100vw,¿cuánto espacio ocupa la imagen?
         (max-width: 900px) 50vw,
         33vw"
  alt="Descripción de la imagen"
>
HTML<!-- srcset con descriptor de ancho (w) + sizes --> <img src="estadio-800.jpg" srcset="estadio-400.jpg 400w, estadio-800.jpg 800w, estadio-1200.jpg 1200w, estadio-1600.jpg 1600w" sizes="(max-width: 600px) 100vw, (max-width: 900px) 50vw, 33vw" alt="Vista aérea del Estadio Azteca con capacidad para 87,000 personas" loading="lazy" > <!-- Cómo lo lee el navegador: 1. Mira el ancho de la pantalla 2. Consulta "sizes" para saber cuánto espacio ocupa la imagen ahí 3. Multiplica por el DPR (pixel ratio del dispositivo) 4. Elige la imagen de srcset más adecuada -->

Descriptores de densidad (x) — para pantallas retina

HTML<!-- Para íconos y logos donde el tamaño no cambia pero la nitidez sí --> <img src="logo.png" srcset="logo.png 1x, logo-2x.png 2x, logo-3x.png 3x" alt="Logo del equipo" width="120" height="40" > <!-- En iPhone con Retina (2x), carga logo-2x.png automáticamente --> <!-- En pantallas Pro Max (3x), carga logo-3x.png -->

💡 ¿Qué son el DPR y las pantallas Retina?

El Device Pixel Ratio (DPR) es cuántos píxeles físicos tiene el dispositivo por cada píxel CSS. Un iPhone con pantalla Retina tiene DPR=2: necesita una imagen de 400px de ancho para mostrar nítidamente un elemento de 200px CSS. Sin srcset, el navegador estira la imagen de 200px a 400px físicos y se ve borrosa.

🎨
<picture> — Control total sobre qué imagen cargar
Diferentes formatos, diferentes composiciones, un solo elemento HTML

Mientras que srcset sirve la misma imagen en diferentes resoluciones, <picture> te permite servir imágenes completamente diferentes según las condiciones. Puedes cambiar el formato (WebP vs JPEG), la composición (imagen recortada diferente para móvil), o cualquier otra condición.

<picture>
  /* El navegador prueba de arriba a abajo y usa el primero que puede */
  <source media="(min-width: 800px)" srcset="estadio-landscape.jpg">Desktop: imagen horizontal
  <source media="(min-width: 400px)" srcset="estadio-cuadrado.jpg">Tablet: imagen cuadrada
  <img src="estadio-vertical.jpg" alt="Estadio">Móvil: imagen vertical (fallback)
</picture>

El caso de uso más importante: WebP con fallback JPEG

WebP es un formato de imagen moderno creado por Google que pesa entre un 25% y 35% menos que JPEG con la misma calidad. Todos los navegadores modernos lo soportan, pero para garantizar compatibilidad puedes usar <picture> con un fallback:

HTML<picture> <!-- Primero: formato moderno WebP (más ligero) --> <source type="image/webp" srcset="foto-400.webp 400w, foto-800.webp 800w, foto-1600.webp 1600w" sizes="(max-width: 600px) 100vw, 50vw" > <!-- Fallback: JPEG para navegadores viejos --> <source type="image/jpeg" srcset="foto-400.jpg 400w, foto-800.jpg 800w, foto-1600.jpg 1600w" sizes="(max-width: 600px) 100vw, 50vw" > <!-- img: siempre obligatorio como último fallback --> <img src="foto-800.jpg" alt="Jugadores del América celebrando en el Estadio Azteca" loading="lazy" width="800" height="450" > </picture>

Art Direction — composición diferente para cada dispositivo

HTML<!-- En desktop: foto panorámica del estadio completo --> <!-- En móvil: zoom en el jugador (no se ve nada en la foto panorámica pequeña) --> <picture> <source media="(min-width: 768px)" srcset="panoramica-estadio.jpg"> <source media="(min-width: 400px)" srcset="plano-medio.jpg"> <img src="zoom-jugador.jpg" alt="Haaland celebrando gol"> </picture>

🟢 Resumen: cuándo usar qué

  • srcset con w + sizes: misma imagen, diferentes resoluciones. El caso más común.
  • srcset con x: logos e íconos con densidad para pantallas Retina.
  • picture + source type: diferentes formatos (WebP + JPEG fallback).
  • picture + source media: art direction — composición diferente para cada pantalla.

Media Queries — CSS condicional según el dispositivo
Aplicar estilos solo cuando se cumple una condición del entorno

Una media query es como un if para CSS: "si la pantalla mide menos de 600px, aplica estos estilos". Te permite tener reglas CSS que solo se activan bajo ciertas condiciones del entorno del usuario.

🌡️ Analogía: el termostato inteligente

Un termostato inteligente no tiene una sola temperatura: dice "si es verano y más de 28°C, pon el aire acondicionado; si es invierno y menos de 18°C, pon la calefacción". Las media queries hacen lo mismo con CSS: "si la pantalla es pequeña, quita el sidebar; si es grande, muéstralo de nuevo".

Anatomía de una media query

@media screen and (max-width: 768px) {
  /* Estilos que solo aplican cuando se cumple la condición */
  .sidebar { display: none; }
}

@media tipo de medio operador condición

Los tipos de condición más usados

CSS/* ── ANCHO ── */ @media (max-width: 768px) { } /* Pantallas de 768px o menos */ @media (min-width: 768px) { } /* Pantallas de 768px o más */ @media (width: 768px) { } /* Exactamente 768px (raro) */ /* Rango — pantallas entre 600px y 900px */ @media (min-width: 600px) and (max-width: 900px) { } /* Sintaxis moderna de rango (CSS Media Queries Level 4) */ @media (600px <= width <= 900px) { } /* ── ORIENTACIÓN ── */ @media (orientation: portrait) { } /* El dispositivo está vertical */ @media (orientation: landscape) { } /* El dispositivo está horizontal */ /* ── PREFERENCIAS DEL USUARIO ── */ @media (prefers-color-scheme: dark) { } /* Modo oscuro del sistema */ @media (prefers-color-scheme: light) { } /* Modo claro */ @media (prefers-reduced-motion: reduce) { } /* El usuario desactivó animaciones */ @media (prefers-contrast: high) { } /* El usuario necesita más contraste */ /* ── TIPO DE MEDIO ── */ @media screen { } /* Pantallas */ @media print { } /* Al imprimir — oculta navbars, fondos, etc. */ @media all { } /* Todos (por defecto si no especificas) */ /* ── OPERADORES LÓGICOS ── */ @media (min-width: 600px) and (orientation: landscape) { } /* Ambas condiciones */ @media (max-width: 400px), (orientation: portrait) { } /* Al menos una condición */ @media not (prefers-color-scheme: dark) { } /* Negación */

Modo oscuro con prefers-color-scheme

CSS/* Define colores base (modo claro) */ :root { --bg: #ffffff; --text: #1a1814; --surface: #f7f4ef; --border: #e8e4de; } /* Sobreescribe automáticamente si el sistema está en modo oscuro */ @media (prefers-color-scheme: dark) { :root { --bg: #0f0f0f; --text: #f0ece6; --surface: #1a1a1a; --border: #2a2a2a; } } /* ¡Con CSS variables, toda la página cambia automáticamente! */ body { background: var(--bg); color: var(--text); }

Media queries para impresión

CSS/* Cuando el usuario imprime la página (Ctrl+P) */ @media print { /* Ocultar elementos innecesarios en impresión */ nav, header, footer, .sidebar, .modal, .btn, video { display: none; } /* Mostrar URLs de los links (el usuario no puede hacer clic en papel) */ a::after { content: ' (' attr(href) ')'; font-size: .8em; color: #666; } /* Fondo blanco, texto negro — tinta ahorro */ body { background: white; color: black; } /* Evitar que los artículos se corten en dos páginas */ .articulo { break-inside: avoid; } }

Breakpoints — Los puntos de quiebre del diseño
En qué anchos cambiar el layout y cuáles son los estándares

💡 Los breakpoints más usados en proyectos reales

En la mayoría de proyectos con solo 3 breakpoints es suficiente: 480px (móvil → phablet), 768px (tablet → desktop pequeño) y 1200px (desktop → wide). No necesitas cubrir todos los tamaños posibles: cubre los puntos donde tu diseño específico "se rompe".

CSS/* Definir breakpoints como variables CSS para consistencia */ :root { --bp-sm: 480px; --bp-md: 768px; --bp-lg: 1024px; --bp-xl: 1280px; } /* Nota: las variables CSS NO funcionan dentro de @media — son solo para documentar */ /* Los breakpoints típicos en uso */ @media (min-width: 480px) { /* sm */ } @media (min-width: 768px) { /* md */ } @media (min-width: 1024px) { /* lg */ } @media (min-width: 1280px) { /* xl */ }

Mobile First — La estrategia correcta
Diseñar primero para móvil y expandir hacia desktop

Mobile First es la estrategia de escribir los estilos base para móvil primero, y luego usar media queries con min-width para ir agregando estilos a medida que la pantalla crece. Es lo opuesto de Desktop First, que era el enfoque antiguo.

🏠 Analogía: construir una casa vs un rascacielos

Desktop First es como construir un rascacielos y luego intentar convertirlo en casa: tienes que quitar muchas cosas. Mobile First es construir una casa sólida y luego agregarle pisos: es mucho más natural y eficiente. Empiezas con lo esencial (móvil) y vas agregando complejidad (desktop).

❌ Desktop First (forma antigua)
/* Estilos para desktop (base) */ .nav { display: flex; } .sidebar { width: 260px; } .cards { grid-template-columns: repeat(3,1fr); } /* ROMPIENDO todo para móvil — max-width */ @media (max-width: 768px) { .nav { display: none; } .sidebar { display: none; } .cards { grid-template-columns: 1fr; } }
✅ Mobile First (forma moderna)
/* Estilos base para móvil — min-width */ .nav { display: none; } .sidebar { display: none; } .cards { display: flex; flex-direction: column; } /* MEJORANDO para pantallas más grandes */ @media (min-width: 768px) { .nav { display: flex; } .sidebar { display: block; } .cards { flex-direction: row; flex-wrap: wrap; } }

Ejemplo completo — Navbar Mobile First

CSS/* ── BASE: Móvil ── */ .navbar { display: flex; justify-content: space-between; align-items: center; padding: .75rem 1rem; } .navbar__links { display: none; } /* Oculto en móvil */ .navbar__hamburger { display: block; } /* Visible en móvil */ .grid { grid-template-columns: 1fr; } /* 1 columna en móvil */ /* ── TABLET (768px) ── */ @media (min-width: 768px) { .grid { grid-template-columns: repeat(2, 1fr); } /* 2 columnas */ } /* ── DESKTOP (1024px) ── */ @media (min-width: 1024px) { .navbar__links { display: flex; } /* Links visibles */ .navbar__hamburger{ display: none; } /* Sin hamburguesa */ .grid { grid-template-columns: repeat(3, 1fr); } /* 3 columnas */ } /* ── WIDE (1280px) ── */ @media (min-width: 1280px) { .grid { grid-template-columns: repeat(4, 1fr); } /* 4 columnas */ }

Container Queries — El futuro del responsive
Estilos basados en el tamaño del contenedor, no de la pantalla

Las Container Queries son la evolución de las media queries. En vez de preguntar "¿cuánto mide la pantalla?", preguntan "¿cuánto mide el contenedor de este componente?". Esto hace que los componentes sean verdaderamente reutilizables: se adaptan a donde los pongas, no a la pantalla global.

🧩 Analogía: las piezas de Lego vs el cuarto

Con media queries, las piezas de Lego (componentes) cambian de forma según el tamaño del cuarto (pantalla). Esto está bien, pero ¿qué pasa si pones la misma pieza en un cuarto grande en un rincón pequeño? La pieza no sabe que está en un espacio pequeño. Las Container Queries hacen que la pieza se adapte al espacio que ocupa, no al cuarto entero.

CSS/* 1. Declarar el contenedor */ .tarjeta-wrapper { container-type: inline-size; /* Activa container queries en el ancho */ container-name: tarjeta; /* Nombre opcional para referenciar */ } /* 2. Usar @container en vez de @media */ @container tarjeta (min-width: 380px) { /* Cuando el CONTENEDOR tiene 380px o más, cambiar el layout */ .tarjeta { flex-direction: row; /* Imagen al lado del texto */ } } @container tarjeta (min-width: 560px) { .tarjeta__titulo { font-size: 1.1rem; } }

🟢 Container Queries vs Media Queries — ¿cuándo usar cuál?

  • Media Queries: cambios de layout de página completa (sidebar que aparece, navbar que cambia, número de columnas del grid principal)
  • Container Queries: componentes que deben adaptarse al espacio que ocupan (tarjetas, cards de producto, widgets, componentes reutilizables)
  • En proyectos modernos se usan ambas juntas: media queries para el layout macro, container queries para los componentes micro.

@supports — Feature Queries
Aplicar estilos solo si el navegador soporta una propiedad CSS

Las Feature Queries con @supports son como media queries pero para características CSS: "si el navegador soporta esta propiedad, aplica estos estilos; si no, usa el fallback". Permiten usar CSS moderno de forma progresiva.

🔌 Analogía: el adaptador de corriente universal

Cuando viajas, llevas un adaptador universal porque no sabes qué tipo de enchufes habrá en el destino. @supports es ese adaptador: "si hay enchufe tipo A, úsalo; si no, busca el tipo B". Tu página funciona en todos lados, pero usa lo mejor disponible en cada navegador.

CSS/* Usar CSS Grid si el navegador lo soporta, con fallback en Flexbox */ /* Fallback (todos los navegadores): */ .galeria { display: flex; flex-wrap: wrap; gap: 1rem; } /* Mejora progresiva — solo si Grid está soportado: */ @supports (display: grid) { .galeria { display: grid; grid-template-columns: repeat(auto-fit, minmax(200px, 1fr)); } } /* Operadores lógicos */ @supports (display: grid) and (gap: 1rem) { } @supports (display: grid) or (display: flex) { } @supports not (display: grid) { } /* Casos de uso modernos */ @supports (backdrop-filter: blur(10px)) { .header { backdrop-filter: blur(10px); background: rgba(255,255,255,.7); } } @supports (container-type: inline-size) { .wrapper { container-type: inline-size; } } @supports selector(:has(img)) { /* Solo si :has() está soportado */ .card:has(img) { padding: 0; } }

💡 Mejora progresiva (Progressive Enhancement)

La filosofía detrás de @supports es la mejora progresiva: empieza con una base que funcione en todos lados, y luego usa características modernas para mejorar la experiencia donde estén disponibles. Lo opuesto es la "degradación elegante" (empezar con lo más moderno y agregar fallbacks). Mobile First y Progressive Enhancement van de la mano.

Ejercicio integrador — Sitio de noticias deportivas responsive
Mobile first, imágenes adaptables, media queries y container queries
✏️ Ejercicio — Página de noticias deportivas 100% responsive Avanzado

Crea noticias-deportivas.html con estrategia Mobile First. El sitio debe verse perfecto en celular, tablet y desktop.

  1. Estructura base (móvil): Todo en columna única. La navbar muestra solo el logo y un ícono de menú (hamburguesa). El hero ocupa el 100% del ancho. Las tarjetas de noticias están en una sola columna.
  2. Imágenes responsive con <picture>: Cada tarjeta de noticia debe tener una imagen con al menos dos versiones: WebP como primera opción y JPEG como fallback. Usa loading="lazy", width y height definidos.
  3. Media query tablet (768px): Las tarjetas pasan a una grilla de 2 columnas. La navbar muestra los links de navegación. El hero pasa a ser horizontal (imagen a la derecha, texto a la izquierda con Flexbox).
  4. Media query desktop (1024px): Las tarjetas pasan a 3 columnas. Aparece un sidebar lateral con las noticias más leídas. El layout es sidebar (260px fijo) + contenido principal (flex: 1).
  5. Container Query en las tarjetas: Cada tarjeta debe usar container-type: inline-size. Cuando la tarjeta tenga más de 400px de ancho (como en el sidebar o en desktop wide), la imagen debe aparecer al lado del texto en vez de arriba.
  6. Modo oscuro: Con @media (prefers-color-scheme: dark) y variables CSS, el fondo, texto y superficies cambian automáticamente.
  7. @supports: Usa @supports (backdrop-filter: blur()) para darle un efecto glassmorphism a la navbar en navegadores modernos, con un fondo sólido como fallback.

Bonus: Agrega un botón "Cambiar tema" que con JavaScript agrega/quita la clase .dark al <html>, y usa :root.dark { } en el CSS para sobreescribir las variables. Así el usuario puede elegir el modo independientemente de la preferencia del sistema.

🟢 Resumen de lo que aprendiste en la Parte F

  • Responsive Design: grillas fluidas + imágenes flexibles + media queries
  • Meta viewport: obligatorio, nunca user-scalable=no

  • srcset con w: múltiples resoluciones + sizes para que el navegador elija
  • srcset con x: densidad de pantalla para íconos y logos

  • @media: min-width, max-width, orientation, prefers-color-scheme, prefers-reduced-motion, print

  • Mobile First: estilos base para móvil, min-width para expandir
  • Container Queries (@container): componentes que se adaptan a su contenedor
  • Feature Queries (@supports): mejora progresiva con fallbacks
