Metatags — Información invisible pero crucial
Lo que el navegador y Google leen, pero tú no ves en la página

Las metatags van dentro del <head> y le dicen cosas importantes al navegador, a los motores de búsqueda (Google, Bing) y a las redes sociales (WhatsApp, Twitter) sobre tu página. El usuario nunca las ve directamente, pero tienen un efecto enorme en cómo se muestra y comparte tu sitio.

📦 Analogía: la etiqueta de un paquete de envío

Cuando envías un paquete, la caja tiene una etiqueta con el remitente, el destinatario, el contenido, si es frágil, etc. Nadie que recibe el paquete lee esa etiqueta para saber qué hay dentro del paquete en sí, pero el servicio de mensajería la usa para enrutar y manejar el paquete correctamente. Las metatags son exactamente eso: la etiqueta de envío de tu página web.

Anatomía de una metatag

<meta name="description" content="Descripción de mi página">
        Etiqueta vacía name = tipo de meta content = el valor

Las metatags son etiquetas vacías (sin cierre). Tienen dos atributos principales: name (qué tipo de información es) y content (el valor de esa información).

Las metatags más importantes

<meta charset="UTF-8">

Codificación de caracteres. Le dice al navegador cómo leer los caracteres del archivo. UTF-8 soporta prácticamente todos los idiomas del mundo, incluyendo tildes (á, é, í), ñ, emojis, etc.

⚠️ Siempre debe ser la primera línea del <head>. Sin esta, las tildes y la ñ aparecen como símbolos raros.

<meta name="viewport">

Viewport responsive. Le dice al navegador en móviles que use el ancho real del dispositivo, no que simule una pantalla de escritorio.

content="width=device-width, initial-scale=1.0"

Sin esto, tu página en celular se verá ridículamente pequeña como si la viera con zoom muy alejado.

<meta name="description">

Descripción para Google. El texto que aparece debajo del título de tu página en los resultados de búsqueda. Máximo ~160 caracteres.

content="Aprende HTML y CSS desde cero con ejemplos prácticos"

Es crucial para SEO. Un buen description convence al usuario de hacer clic en tu enlace.

<meta name="author">

Autor del documento. Declara quién escribió la página. Lo usan principalmente los navegadores y herramientas de lectura.

content="Hugo"

No afecta directamente el SEO, pero es buena práctica profesional.

<meta name="robots">

Instrucciones para robots de búsqueda. Le dices a Google si puede o no indexar tu página y seguir sus enlaces.

content="index, follow" — aparece en Google
content="noindex, nofollow" — oculta de Google

Muy útil para páginas en construcción o páginas de administración que no quieres que aparezcan en Google.

<meta http-equiv="refresh">

Redirección automática. Redirige al usuario a otra URL después de N segundos.

content="5; url=https://mipagina.com"

Redirige después de 5 segundos. Con content="0; url=..." redirige inmediatamente.

Open Graph — Metatags para redes sociales

Cuando alguien comparte tu página en WhatsApp, Twitter o Facebook, esas plataformas buscan las metatags Open Graph (og:) para saber cómo mostrar la vista previa: qué imagen mostrar, qué título y qué descripción usar.

HTML<!-- Open Graph — controla la vista previa en redes sociales --> <meta property="og:title" content="Mi Portafolio Web"> <meta property="og:description" content="Desarrollador web especializado en HTML y CSS"> <meta property="og:image" content="https://mipagina.com/preview.png"> <meta property="og:url" content="https://mipagina.com"> <meta property="og:type" content="website">

🟢 ¿Cuándo es importante Open Graph?

Si estás construyendo un blog, un portafolio o cualquier página que la gente pueda compartir, el Open Graph es esencial. Sin él, cuando alguien comparte tu URL en WhatsApp, aparece sin imagen, con un título horrible y sin descripción.

Ejemplo completo del <head> con metatags profesional

HTML<head> <!-- ① SIEMPRE primero: codificación --> <meta charset="UTF-8"> <!-- ② Responsive en móvil --> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <!-- ③ Título que aparece en pestaña y en Google --> <title>Hugo Dev — Portafolio Web</title> <!-- ④ SEO básico --> <meta name="description" content="Portafolio de proyectos web de Hugo, especializado en HTML, CSS y JavaScript."> <meta name="author" content="Hugo"> <!-- ⑤ Open Graph para redes sociales --> <meta property="og:title" content="Hugo Dev — Portafolio Web"> <meta property="og:description" content="Proyectos de desarrollo web: HTML, CSS, JS."> <meta property="og:image" content="preview.png"> <meta property="og:type" content="website"> <!-- ⑥ CSS externo --> <link rel="stylesheet" href="estilos.css"> </head>

Comentarios en HTML
Notas para ti (y tu futuro yo) que el navegador ignora

Un comentario en HTML es texto que puedes escribir dentro del código sin que aparezca en la página. Solo lo ves tú (y cualquiera que abra el código fuente). Se escribe entre <!-- y -->.

📓 Analogía: las notas en los márgenes de un libro

Cuando estudias, escribes notas al margen del libro para recordar cosas: "esto es importante", "revisar más tarde", "esto lo explica el profesor de otra forma". Esas notas son para ti, no forman parte del texto del libro. Los comentarios HTML son exactamente eso: tus notas al margen del código.

Sintaxis del comentario

HTML<!-- Esto es un comentario en una línea --> <!-- Esto es un comentario que ocupa varias líneas. Muy útil para explicar bloques grandes de código. --> <header> <!-- Aquí va el logo y la navegación --> <img src="logo.png" alt="Logo"> </header> <!-- TODO: agregar menú hamburguesa para móvil --> <!-- IMPORTANTE: este div tiene que ir antes del main -->

🟢 Usos prácticos de los comentarios

  • Separar secciones: <!-- ═══ HEADER ═══ -->
  • Dejar notas pendientes: <!-- TODO: agregar slider -->
  • Desactivar código temporalmente sin borrarlo
  • Explicar código complejo para cuando lo releas en 6 meses

⚠️ Nunca guardes información sensible en comentarios

Los comentarios son visibles para cualquiera que haga clic derecho → "Ver código fuente" en el navegador. Nunca escribas contraseñas, tokens, claves de API ni información privada en comentarios HTML.

Atajo de teclado en VS Code

Para comentar/descomentar rápidamente en VS Code: selecciona el texto o línea y presiona Ctrl + / (Windows/Linux) o Cmd + / (Mac). ¡Funciona en HTML, CSS y JavaScript!

Íconos — El favicon de tu página
El pequeño ícono que aparece en la pestaña del navegador

El favicon (abreviación de "favorite icon") es el pequeño ícono que ves en la pestaña del navegador junto al título de la página. También aparece cuando guardas un sitio en marcadores. Se agrega dentro del <head> con una etiqueta <link>.

🏪 Analogía: el logo en el letrero de una tienda

Cuando tienes muchas pestañas abiertas en tu navegador, el favicon es como el logo en el letrero de una tienda en un centro comercial: te permite identificar rápidamente cuál pestaña es cuál sin leer el título completo.

Cómo agregar un favicon

HTML<head> <!-- Favicon clásico (archivo .ico) --> <link rel="icon" type="image/x-icon" href="favicon.ico"> <!-- Favicon moderno con PNG (más recomendado hoy en día) --> <link rel="icon" type="image/png" href="favicon.png"> <!-- Favicon SVG (vectorial, escala perfectamente en cualquier tamaño) --> <link rel="icon" type="image/svg+xml" href="favicon.svg"> <!-- Apple Touch Icon (el ícono cuando guardas en pantalla de inicio en iPhone) --> <link rel="apple-touch-icon" href="apple-icon.png"> </head>

🟢 Dónde generar tu favicon gratis

  • favicon.io — genera un favicon desde texto, emoji o imagen. Muy fácil.
  • realfavicongenerator.net — genera los favicons para todos los dispositivos a la vez.
  • El tamaño estándar es 32×32 px para el ícono de pestaña, y 180×180 px para Apple Touch Icon.

Íconos con Font Awesome (íconos como fuente)

Una técnica muy popular es usar librerías de íconos SVG como Font Awesome. En lugar de imágenes, los íconos son vectores que escalan perfecto y se estilizan con CSS.

HTML<!-- 1. Agrega el CDN en el head --> <link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.5.0/css/all.min.css"> <!-- 2. Úsalos con la etiqueta <i> y clases de Font Awesome --> <i class="fa-solid fa-house"></i> <!-- 🏠 ícono de casa --> <i class="fa-solid fa-envelope"></i> <!-- ✉️ ícono de correo --> <i class="fa-brands fa-github"></i> <!-- 🐙 logo de GitHub --> <i class="fa-brands fa-linkedin"></i> <!-- 💼 logo de LinkedIn --> <!-- Los íconos heredan el color y tamaño del CSS como texto normal --> <i class="fa-solid fa-star" style="color: gold; font-size: 2rem;"></i>

💡 ¿Por qué usar <i> para íconos?

La etiqueta <i> originalmente era para texto en cursiva (italic). Con Font Awesome se "reutilizó" para íconos porque es corta y se puede dejar vacía. No tiene significado semántico propio cuando se usa así, lo cual está bien para elementos puramente decorativos.

Textarea — Área de texto multilínea
Para mensajes, comentarios y cualquier texto largo

Ya conoces <input type="text"> para una sola línea de texto. Pero ¿qué pasa cuando el usuario necesita escribir un párrafo completo, un mensaje, una descripción? Para eso existe <textarea>: un campo de texto de múltiples líneas que el usuario puede redimensionar.

📄 Analogía: una hoja en blanco vs una tarjeta

Un <input type="text"> es como una tarjeta de presentación: espacio pequeño, una línea, solo cabe tu nombre y teléfono. Un <textarea> es como una hoja en blanco: puedes escribir todo lo que necesitas, cuántas líneas quieras.

Anatomía del textarea

<textarea name="mensaje" rows="5" cols="40" placeholder="Escribe tu mensaje..." maxlength="500"> Texto inicial aquí </textarea>
No es etiqueta vacía rows = líneas visibles cols = ancho en caracteres

⚠️ Diferencia clave con <input>

<textarea> NO es una etiqueta vacía. Tiene etiqueta de apertura y cierre. El contenido que escribas entre las etiquetas aparecerá como texto inicial en el campo.

Atributos importantes del textarea

AtributoQué haceEjemplo
rowsNúmero de líneas visibles (alto del área)rows="5"
colsAncho en número de caracteres (usa CSS en su lugar)cols="40"
placeholderTexto gris de ejemplo que desaparece al escribirplaceholder="Tu mensaje..."
maxlengthNúmero máximo de caracteres permitidosmaxlength="500"
minlengthNúmero mínimo de caracteres requeridosminlength="10"

Controlar el resize con CSS

Por defecto el usuario puede arrastrar la esquina del textarea para cambiar su tamaño. Puedes controlarlo con CSS:

CSS/* El usuario puede redimensionar en todas direcciones (por defecto) */ textarea { resize: both; } /* Solo puede cambiar el alto (vertical) — lo más común */ textarea { resize: vertical; } /* No puede redimensionar en absoluto */ textarea { resize: none; }

🟢 Buenas prácticas con textarea

  • Usa resize: vertical para que el usuario solo pueda cambiar el alto, no el ancho (evita que rompa el layout)
  • Usa font-family: inherit en CSS para que el textarea use la misma fuente que el resto de la página (por defecto usa monospace)
  • Siempre usa un <label> asociado (lo vemos a continuación)

Label — La etiqueta de los campos de formulario
Accesibilidad y usabilidad: el compañero inseparable de los inputs

El <label> es el texto descriptivo que va junto a un campo de formulario. No es solo decoración: cuando lo vinculas correctamente al input, hacer clic en el label activa el input. Es fundamental para la accesibilidad (lectores de pantalla) y la usabilidad.

📋 Analogía: el letrero de un cajón

Imagina un mueble con cajones sin letrero. Tienes que abrir cada uno para saber qué hay adentro. Ahora imagina los mismos cajones con letreros: "Documentos", "Facturas", "Contratos". El label es ese letrero: le dice al usuario exactamente para qué sirve ese campo.

Dos formas de vincular label con input

Forma 1 — for + id (recomendada)

HTML<!-- El atributo "for" del label debe coincidir con el "id" del input --> <label for="nombre">Tu nombre completo</label> <input type="text" id="nombre" name="nombre" placeholder="Ej: Hugo Castillo"> <!-- Al hacer clic en "Tu nombre completo", el cursor va al input automáticamente -->

Forma 2 — label envolviendo al input

HTML<!-- El input va DENTRO del label. No necesitas for ni id --> <label> Tu nombre completo <input type="text" name="nombre" placeholder="Ej: Hugo Castillo"> </label>

💡 ¿Cuál forma usar?

La Forma 1 (for + id) es más flexible porque el label y el input pueden estar en distintas partes del HTML. La Forma 2 es más compacta pero el input tiene que estar dentro del label. Ambas son correctas. En la práctica, la Forma 1 es más común en diseños profesionales.

Label con checkboxes y radios — caso especial

Los labels son especialmente importantes con checkboxes y radios. Sin un label vinculado, el usuario tiene que hacer clic exactamente en el pequeño cuadro/círculo. Con un label, puede hacer clic en cualquier parte del texto:

HTML<!-- Sin label: solo el cuadrito activa el checkbox --> <input type="checkbox"> Acepto los términos <!-- ❌ el texto no es clickeable --> <!-- Con label: el cuadrito Y el texto activan el checkbox --> <input type="checkbox" id="terminos"> <label for="terminos">Acepto los términos</label> <!-- ✅ todo el texto es clickeable --> <!-- Botones radio con labels --> <input type="radio" id="html" name="lenguaje" value="html"> <label for="html">HTML</label> <input type="radio" id="css" name="lenguaje" value="css"> <label for="css">CSS</label>

Select, Datalist y Option
Menús desplegables y listas de sugerencias

Cuando quieres que el usuario elija entre opciones predefinidas, tienes dos herramientas principales: <select> (lista desplegable cerrada) y <datalist> (sugerencias mientras escribe). Cada una tiene su caso de uso.

<select> — El menú desplegable clásico

🎰 Analogía: la rueda de selección de una máquina de café

Una máquina expendedora de café tiene una rueda con opciones: Americano, Capuchino, Latte, Espresso. Solo puedes elegir las opciones que están ahí, no puedes inventarte una. Eso es exactamente un <select>: un conjunto fijo de opciones del cual el usuario elige una (o varias).

<select name="pais" id="pais">
  <option value="" disabled selected>-- Selecciona un país --</option>
  <option value="mx">México</option>
  <option value="es">España</option>
</select>
Contenedor del menú value = lo que se envía al servidor texto = lo que ve el usuario

ElementoAtributoQué hace
<select>multiplePermite seleccionar varias opciones a la vez (Ctrl+clic)
<select>size="3"Muestra N opciones visibles sin necesidad de abrir el menú
<select>requiredObliga a seleccionar una opción antes de enviar
<option>valueEl valor que se envía al servidor (puede ser diferente al texto)
<option>selectedEsta opción aparece seleccionada por defecto

Agrupar opciones con <optgroup>

Datalist — El buscador con sugerencias
Lo mejor de los dos mundos: escribir libremente y recibir sugerencias

El <datalist> es diferente al <select>. Mientras que el select obliga a elegir de una lista cerrada, el datalist le muestra sugerencias al usuario mientras escribe, pero le permite escribir algo diferente si quiere.

🔍 Analogía: el buscador de Google

Cuando escribes en el buscador de Google, aparecen sugerencias debajo, pero tú puedes ignorarlas y escribir exactamente lo que quieras. Eso es un datalist: sugerencias, no imposiciones.

¿Cómo funciona?

Necesitas dos elementos trabajando juntos: un <input> y un <datalist>. Los conectas con el atributo list del input y el id del datalist.

HTML<label for="equipo">Tu equipo de fútbol favorito:</label> <!-- 1. El input tiene list="nombre-del-datalist" --> <input type="text" id="equipo" name="equipo" list="equipos-lista" placeholder="Escribe o elige un equipo..." > <!-- 2. El datalist tiene el id que coincide con el list del input --> <datalist id="equipos-lista"> <option value="Real Madrid"> <option value="Barcelona"> <option value="América"> <option value="Chivas"> <option value="Cruz Azul"> <option value="Monterrey"> <option value="Tigres"> </datalist>

Escribe "A" o "C" para ver las sugerencias filtradas

Fieldset y Legend — Agrupar campos del formulario
Organización visual y semántica de grupos de campos relacionados

Cuando un formulario tiene muchos campos, es buena práctica agrupar los relacionados. <fieldset> crea un contenedor visual con borde alrededor de un grupo de campos, y <legend> es el título de ese grupo que aparece incrustado en el borde.

🗂 Analogía: las secciones de un formulario de papel

Cuando llegas a un hospital a llenar un formulario, el papel tiene secciones claramente delimitadas: "Datos personales" (con un rectángulo y título), "Datos médicos" (otro rectángulo), "Datos del seguro" (otro). Eso es exactamente fieldset + legend: un rectángulo con título para agrupar campos relacionados.

Estructura de fieldset y legend

HTML<form> <fieldset> <legend>👤 Datos personales</legend> <label for="nombre">Nombre completo</label> <input type="text" id="nombre" name="nombre"> <label for="email">Correo electrónico</label> <input type="email" id="email" name="email"> </fieldset> <fieldset> <legend>🔒 Datos de acceso</legend> <label for="usuario">Usuario</label> <input type="text" id="usuario" name="usuario"> <label for="pass">Contraseña</label> <input type="password" id="pass" name="pass"> </fieldset> </form>
👤 Datos personales
🔒 Datos de acceso

🟢 Fieldset también ayuda a la accesibilidad

Los lectores de pantalla (para personas con discapacidad visual) anuncian el <legend> cuando el usuario navega con el teclado hasta ese grupo de campos. Así la persona sabe en qué sección del formulario está sin verla visualmente.

Deshabilitar un grupo completo de campos

Un superpower de <fieldset>: si le pones el atributo disabled, todos los campos dentro quedan desactivados de un solo golpe. No tienes que ir campo por campo.

HTML<!-- Deshabilitar todos los campos del grupo de pago en un clic --> <fieldset disabled> <legend>💳 Datos de pago (sección no disponible)</legend> <input type="text" placeholder="Número de tarjeta"> <input type="text" placeholder="Fecha de vencimiento"> </fieldset>

Details y Summary — Acordeón nativo de HTML
Contenido expandible/colapsable sin necesidad de JavaScript

¿Alguna vez has visto esas secciones de "Preguntas Frecuentes" donde haces clic en la pregunta y se expande la respuesta? En muchos sitios eso se hace con JavaScript, pero HTML tiene un elemento nativo para esto: <details> con <summary>. Cero JavaScript necesario.

📪 Analogía: el buzón de correo

Un buzón de correo tiene una tapa cerrada. Ves el buzón pero no el contenido. Cuando levantas la tapa (clic en el summary), ves todo lo que hay adentro. Cuando la cierras, el contenido vuelve a estar oculto. Eso es <details>: un contenedor con tapa (<summary>) que el usuario puede abrir y cerrar.

Estructura básica

HTML<!-- Cerrado por defecto --> <details> <summary>¿Cuánto tiempo tarda aprender HTML?</summary> <!-- Todo lo que esté aquí aparece al hacer clic en el summary --> <p>Con práctica diaria, puedes aprender lo básico en 2-4 semanas.</p> </details> <!-- Abierto por defecto con el atributo "open" --> <details open> <summary>¿Necesito JavaScript para hacer páginas web?</summary> <p>Para páginas estáticas, no. HTML y CSS son suficientes para sitios informativos, portafolios y páginas de presentación.</p> </details>
¿Cuánto tiempo tarda aprender HTML?

Con práctica diaria, puedes aprender lo básico en 2–4 semanas. La clave es practicar construyendo proyectos reales, no solo leer teoría.

¿Necesito JavaScript para hacer páginas web?

Para páginas estáticas, no. HTML y CSS son suficientes para sitios informativos, portafolios y páginas de presentación. JavaScript agrega interactividad dinámica.

¿Puedo poner imágenes dentro de details?

¡Sí! Puedes poner cualquier contenido HTML dentro de details: párrafos, listas, tablas, imágenes, incluso otros details anidados.

Personalizar el triángulo con CSS

Por defecto, el navegador muestra un triángulo ▶ antes del summary. Puedes quitarlo o reemplazarlo:

CSS/* Quitar el triángulo por defecto */ summary { list-style: none; cursor: pointer; } summary::-webkit-details-marker { display: none; } /* Safari */ /* Agregar tu propio ícono */ summary::before { content: '+'; margin-right: .5rem; font-weight: bold; color: var(--primary); } details[open] summary::before { content: '−'; /* Cambia a − cuando está abierto */ }

🟢 ¿Cuándo usar details/summary vs JavaScript?

  • Usa details/summary cuando quieres algo simple: FAQ, notas adicionales, contenido optativo. Sin JavaScript, accesible por defecto, funciona en todos los navegadores modernos.
  • Usa JavaScript cuando necesitas animaciones complejas, transiciones suaves, o lógica especial (ej: que al abrir uno se cierren los demás automáticamente — un acordeón exclusivo).

Anidar details — acordeón dentro de acordeón

HTML<details> <summary>📚 Módulo 1: HTML Básico</summary> <ul> <li>Estructura de una página</li> <li>Etiquetas de texto</li> </ul> <!-- Details dentro de details --> <details> <summary>Ver ejercicios de este módulo</summary> <ol> <li>Crea tu primera página web</li> <li>Agrega títulos y párrafos</li> </ol> </details> </details>

Ejercicio integrador — Formulario de Registro Completo
Pon en práctica todos los elementos de esta parte
✏️ Ejercicio — Formulario de Registro de Torneo de Fútbol Intermedio

Vamos a crear un formulario de registro para un torneo de fútbol que use todos los elementos que aprendimos en esta parte. El formulario debe incluir metatags correctas en el head, labels vinculados, fieldsets con legend, un select, un datalist y un details/summary al final con los términos.

  1. Crea un archivo registro-torneo.html con la estructura base correcta. En el <head> agrega:
    • charset, viewport, title
    • Una meta name="description" describiendo el torneo
    • Una meta name="author" con tu nombre
  2. Sección 1 — Datos del jugador (usa <fieldset> + <legend>):
    • Nombre completo (input text + label)
    • Edad (input number + label)
    • Correo electrónico (input email + label)
  3. Sección 2 — Datos del equipo (otro <fieldset>):
    • Nombre del equipo (input text + label)
    • Posición favorita con <select>: Portero, Defensa, Mediocampista, Delantero
    • Equipo de fútbol favorito con <datalist> (al menos 8 equipos de tu gusto)
    • Mensaje adicional con <textarea> (placeholder, maxlength=300)
  4. Términos y condiciones al final del formulario, usando <details> + <summary>. Dentro del details escribe 3 reglas inventadas del torneo.
  5. Checkbox: "He leído y acepto los términos y condiciones" con su <label> correctamente vinculado.
  6. Un botón de envío: <button type="submit">Inscribirme al torneo</button>

Resultado esperado: Un formulario semántico, bien estructurado, con dos secciones fieldset, opciones de select y datalist, textarea y un acordeón de términos al final. No necesitas CSS por ahora — enfócate en el HTML correcto.

🟢 Resumen de lo que aprendiste en esta parte

  • Metatags: charset, viewport, description, author, robots, Open Graph
  • Comentarios: <!-- --> para notas en el código
  • Favicon: el ícono de la pestaña del navegador
  • Textarea: campo de texto multilínea con resize
  • Label: etiqueta vinculada a un input (for + id)
  • Select + Option: menú desplegable de opciones fijas
  • Optgroup: agrupación de opciones en un select
  • Datalist: sugerencias mientras el usuario escribe
  • Fieldset + Legend: agrupación visual de campos de formulario
  • Details + Summary: acordeón nativo sin JavaScript
