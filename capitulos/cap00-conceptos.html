Antes de empezar

Los conceptos, el vocabulario y la lógica que necesitas para entender todo lo que viene después.

💡

¿Qué es programar?

Darle instrucciones a una máquina que no piensa

🤖 Imagina que tienes un robot asistente muy literal — hace exactamente lo que le dices, ni más ni menos. Si le dices "tráeme un vaso de agua" y no hay vasos, se queda paralizado porque no sabe qué hacer en ese caso. Programar es anticipar todos los casos posibles y darle instrucciones para cada uno.

Programar es escribir instrucciones en un lenguaje que la computadora entiende. Esas instrucciones le dicen qué mostrar, qué guardar, qué hacer cuando alguien hace clic en algo, o qué pasará si ocurre un error.

La diferencia con darle instrucciones a una persona es que la computadora no interpreta ni adivina — sigue las instrucciones al pie de la letra. Si escribes algo mal, no hace lo que querías. No se queja, simplemente falla en silencio o muestra un error.

La mentalidad más importante al programar:

No le hablas a alguien inteligente. Le hablas a algo que ejecuta exactamente lo que escribes. La computadora no sabe "lo que quisiste decir" — solo sabe lo que realmente escribiste. Por eso la precisión importa más aquí que en cualquier otro lugar.
💡

En la práctica: cuando una página web hace algo — guardar tu nombre, mostrar una animación, enviar un formulario — hay código atrás que le dice exactamente cómo hacerlo, paso a paso. Ese código lo escribió alguien como tú aprendiendo exactamente esto.

🗣️

Lenguajes de programación

Idiomas que la computadora entiende

🌍 Como los idiomas humanos — español, inglés, japonés — cada lenguaje de programación tiene su propia gramática y vocabulario. Algunos son para construir páginas web, otros para hacer apps, otros para analizar datos. En esta guía aprendes los tres idiomas de la web.

Existen cientos de lenguajes de programación, cada uno diseñado para hacer algo diferente. Para hacer páginas y aplicaciones web, los tres lenguajes que trabajan juntos son HTML, CSS y JavaScript. No son intercambiables — cada uno tiene un trabajo distinto.

HTML La estructura

Define qué elementos existen en la página: botones, títulos, imágenes, párrafos. Es el esqueleto. Sin HTML no hay nada que ver.

CSS El diseño

Define cómo se ve cada elemento: color, tamaño, posición, tipografía. Sin CSS todo es texto negro sobre fondo blanco.

JS El comportamiento

Define qué hace la página cuando el usuario hace algo: clic, escritura, movimiento. Sin JavaScript todo es estático, como una foto.

La analogía completa:

Piensa en un edificio.
HTML es la estructura física — paredes, puertas, ventanas.
CSS es la decoración — pintura, muebles, iluminación.
JavaScript es la electricidad — hace que las cosas prendan, se muevan y respondan.
🏷️

¿Qué significan las siglas HTML?

HyperText Markup Language

📝 Es un lenguaje de marcado — no es un lenguaje de programación tradicional. No "hace cosas", solo describe y organiza el contenido de una página. Como los títulos, subtítulos y notas al pie de un libro bien estructurado.

Cada palabra de las siglas dice algo importante: 

H  =  HyperText   →  texto que puede tener enlaces a otros textos (lo que conocemos como links)
T  =  Text         →  trabaja principalmente con texto y contenido
M  =  Markup       →  "marcado" — usas marcas (etiquetas) para etiquetar el contenido
L  =  Language     →  lenguaje — tiene reglas de gramática que debes seguir

La parte más importante es Markup — lenguaje de marcado. Significa que rodeas el contenido con "marcas" llamadas etiquetas para decirle al navegador qué tipo de contenido es cada cosa.

HTML 
<!-- Así "marcas" el contenido con etiquetas HTML -->
<h1>Esto es un título</h1>
<p>Esto es un párrafo de texto normal.</p>
<button>Esto es un botón</button>

<!-- Sin las marcas, el navegador no sabe qué es qué -->
Esto es un título
Esto es un párrafo de texto normal.
Esto es un botón
💡

¿Por qué "Hyper"Text? Porque en los años 90, cuando se creó HTML, que un texto pudiera tener enlaces a otros textos era algo revolucionario — "hiper" como en "más allá de". Hoy lo damos por hecho, pero fue la idea que inventó la web.

🎨

¿Qué significan las siglas CSS?

Cascading Style Sheets

🎨 Si HTML es el esqueleto, CSS es la ropa y el maquillaje. Toma los elementos que HTML creó y les dice cómo verse: qué color, qué tamaño, dónde estar, qué forma tener. Sin CSS, todas las páginas se verían igual — texto negro, sin estilo.

Cada palabra de las siglas explica cómo funciona: 

C  =  Cascading    →  "en cascada" — los estilos se heredan y se sobreescriben en orden
S  =  Style        →  estilo — define la apariencia visual de cada elemento
S  =  Sheets       →  hojas — generalmente vive en un archivo separado (.css)

La parte más curiosa es Cascading — en cascada. Significa que los estilos fluyen hacia abajo como una cascada: los estilos del elemento padre llegan a los hijos, y si defines el mismo estilo varias veces, el último gana.

CSS 
/* Así se escribe CSS: selector { propiedad: valor } */

h1 {
  color: navy;          /* todos los h1 serán azul marino */
  font-size: 32px;    /* tamaño 32 píxeles */
}

button {
  background: orange;  /* todos los botones serán naranjas */
  color: white;       /* texto blanco */
  border-radius: 8px; /* esquinas redondeadas */
}
💡

En la práctica: cuando ves que todos los botones de una página tienen el mismo color, o que los títulos siempre tienen la misma fuente — eso es CSS aplicando un estilo a todos los elementos del mismo tipo a la vez.

⚙️

¿Qué es JavaScript?

El único lenguaje que los navegadores ejecutan

🎭 HTML y CSS son como un escenario decorado con actores de cartón — se ven bien pero no hacen nada. JavaScript es lo que hace que los actores se muevan, hablen y reaccionen al público. Es el único lenguaje que todos los navegadores del mundo saben ejecutar.

JavaScript no tiene siglas misteriosas — simplemente se llama así. Lo importante es entender su origen: nació en 1995 para hacer páginas web interactivas y hoy es el lenguaje de programación más usado del mundo.

¿JavaScript y Java son lo mismo?

No. Son lenguajes completamente distintos. Solo tienen nombres parecidos por razones de marketing de los años 90. Es como preguntarle a alguien si habla español porque habla español y espagueti suenan parecido — no tienen relación real.

JavaScript vive en el navegador y puede hacer cosas que HTML y CSS no pueden:

JS 
// ✅ Cosas que solo JS puede hacer:

// 1. Guardar y recordar información
let contador = 0

// 2. Reaccionar a acciones del usuario
boton.addEventListener('click', function() {
  contador = contador + 1    ← suma 1 cada vez que se hace clic
})

// 3. Cambiar la página después de que cargó
titulo.textContent = 'Hola, bienvenido'

// 4. Comunicarse con servidores externos
fetch('https://api.ejemplo.com/datos')   ← pedir datos de internet
💡

En la práctica: cuando llegas a una página y ves que algo cambia sin recargar — un contador sube, aparece un mensaje, se filtra una lista — eso es JavaScript corriendo en tu navegador en tiempo real.

🌐

El navegador

El intérprete que convierte código en páginas

📖 Como un lector que toma un manuscrito escrito en un idioma especial y lo convierte en algo que todos pueden ver y usar. El navegador lee tu código y lo traduce en la página visual que el usuario ve en pantalla.

Chrome, Firefox, Safari, Edge — todos son navegadores. Su trabajo principal no es solo "abrir páginas" — es leer archivos de código (HTML, CSS, JavaScript) e interpretarlos para mostrarlos como una página visual. 

Tu código              El navegador lo lee          El usuario ve

index.html      →      Lee las etiquetas      →    Títulos, botones, imágenes
style.css       →      Aplica los estilos     →    Colores, tamaños, posiciones
script.js       →      Ejecuta el código      →    Interactividad, animaciones

Una herramienta que todo programador web debe conocer es la Consola del navegador. Está escondida pero siempre está ahí. Para abrirla: clic derecho → Inspeccionar → Console (o F12).

La Consola — tu ventana al interior del código

Desde la consola puedes ver errores de JavaScript, probar código en vivo y entender qué está pasando cuando algo no funciona. Es como el panel de diagnóstico de un auto — no lo ves todo el tiempo, pero cuando algo falla es lo primero que abres.
💡

En la práctica: en cualquier página web que visites puedes abrir la consola con F12 y ver el código fuente, los errores, y hasta experimentar con JavaScript en tiempo real. Es la herramienta número uno para aprender y depurar.

✏️

El editor de código

Donde escribes el código — no es un Word

🎹 Como la diferencia entre un piano de concierto y un teclado de juguete — los dos reproducen notas, pero el de concierto tiene pedales, teclas sensibles y resonancia que cambian completamente lo que puedes hacer. Un editor de código tiene herramientas que un bloc de notas nunca tendrá.

Puedes escribir código HTML en el bloc de notas y funcionará — pero un editor de código especializado hace que programar sea mucho más fácil y menos propenso a errores.

El editor más usado en el mundo para desarrollo web es Visual Studio Code (VS Code), gratuito y de Microsoft. Las características que lo hacen especial: 

Resaltado de sintaxis  →  cada tipo de código tiene un color diferente
                           (etiquetas en naranja, valores en verde, etc.)

Autocompletado         →  al escribir "but" te sugiere "button" solo
                           ahorra tiempo y evita errores de tipeo

Detección de errores   →  subraya en rojo las líneas con problemas
                           antes de que abras el navegador

Indentación            →  organiza el código con sangría automática
                           para que sea fácil de leer
¿Por qué no usar Word o Google Docs?

Word agrega formato invisible al texto — negritas, fuentes, espaciado especial. El código debe ser texto limpio y puro, sin ningún formato oculto. Un editor de código guarda archivos de texto plano exactamente como los escribes, sin agregar nada extra.
💡

En la práctica: VS Code tiene extensiones que puedes instalar, como "Live Server" — abre tu página en el navegador y la recarga automáticamente cada vez que guardas el archivo. Es como ver los cambios en tiempo real mientras escribes.

📄

Archivos y carpetas de un proyecto

Cómo se organiza el código

🗂️ Como un libro bien editado — cada tipo de contenido tiene su lugar. El texto en un lugar, las imágenes en otro, los índices en otro. Un proyecto web sigue la misma lógica: cada tipo de archivo tiene su carpeta.

Un proyecto web es simplemente una carpeta con archivos adentro. Hay convenciones — formas estándar que la mayoría de proyectos siguen para organizar esos archivos: 

mi-proyecto/
├── index.html          ← la página principal (siempre se llama así)
├── css/
│   └── style.css       ← todos los archivos de estilo van aquí
├── js/
│   └── script.js       ← todos los archivos de JavaScript van aquí
└── images/
    ├── foto.jpg        ← imágenes, íconos, ilustraciones
    └── logo.png

El archivo index.html es especial — es la convención universal para nombrar la página principal. Cuando un navegador abre una carpeta sin que le digas qué archivo abrir, automáticamente busca index.html.

¿Por qué importa la extensión del archivo?

.html → el navegador lo lee como estructura de página
.css → el navegador lo lee como reglas de estilo
.js → el navegador lo lee como código JavaScript

Si nombras un archivo JavaScript con extensión .txt, el navegador no lo ejecutará — solo lo leerá como texto plano. La extensión le dice al sistema cómo tratar ese archivo.
💡

En la práctica: los archivos HTML se conectan entre sí con enlaces. Un archivo HTML le puede "decir" al navegador que también cargue un archivo CSS y un archivo JS usando etiquetas especiales en su interior. Por eso aunque son archivos separados, el navegador los combina en una sola página.

Los errores — los bugs

No son fracasos, son información

🧩 Como resolver un crucigrama — cuando una palabra no cabe, no fallaste, simplemente descubriste que esa no era la correcta. Los errores te dicen exactamente dónde está el problema, que es mucho mejor que no saber qué está mal.

La palabra "bug" (bicho en inglés) viene de los primeros días de las computadoras, cuando una polilla real se metió en los circuitos y causó un fallo. Desde entonces, cualquier error en el código se llama bug.

Hay tres tipos de errores que encontrarás todo el tiempo: 

1. Error de sintaxis     →  escribiste algo que el lenguaje no entiende
                            como una palabra mal escrita en español
                            Ejemplo: escribir "conslt" en lugar de "const"

2. Error de lógica      →  el código corre pero no hace lo que querías
                            como seguir una receta al pie de la letra
                            pero haberla escrito mal desde el principio

3. Error en tiempo de   →  el código estaba bien pero algo inesperado pasó
   ejecución               mientras corría
                            como pedir un archivo que no existe
Lo que hace un buen programador no es no cometer errores

Es saber leer los mensajes de error y encontrar la solución rápido. Todos los programadores — sin importar la experiencia — cometen errores todo el día. La habilidad que se desarrolla con el tiempo es reconocer los patrones y saber dónde buscar.
💡

En la práctica: cuando el navegador muestra un error en la consola, generalmente te dice el número de línea donde ocurrió. Ese número es tu punto de partida — ve ahí primero. El error muchas veces está en esa línea o en la línea de arriba.

La lógica de programación

Pensar en pasos, condiciones y repeticiones

🍳 Como escribir una receta de cocina muy detallada — paso a paso, en orden, con instrucciones para cada caso. "Si no tienes mantequilla, usa aceite. Si el horno está frío, espera 10 minutos. Repite esto 3 veces." Eso es lógica de programación aplicada a cocinar.

La lógica de programación no es matemática difícil. Es simplemente pensar en instrucciones de forma ordenada y precisa. Toda la programación, sin importar el lenguaje, se basa en tres ideas: 

1. Secuencia        →  las instrucciones se ejecutan en orden, de arriba hacia abajo
                        línea 1, luego línea 2, luego línea 3...

2. Condición        →  "si pasa X, haz Y; si no, haz Z"
                        if / else  →  decisiones

3. Repetición       →  "haz esto N veces" o "haz esto mientras X sea verdad"
                        for / while  →  ciclos o bucles

Absolutamente todo lo que hace cualquier programa del mundo — desde un videojuego hasta un banco — es una combinación de estas tres ideas. Nada más.

JS 
// Secuencia — las líneas corren en orden
let temperatura = 35

// Condición — toma una decisión
if (temperatura > 30) {
  mostrar('Hace calor')
} else {
  mostrar('Temperatura normal')
}

// Repetición — hace algo varias veces
for (let i = 0; i < 5; i++) {
  mostrar('Vuelta número ' + i)
}
Antes de escribir código, piénsalo en español

Si no puedes explicar en palabras lo que quiere hacer tu programa, no podrás escribirlo en código tampoco. Los programadores experimentados primero "escriben" el programa en su cabeza o en papel — en lenguaje humano — y luego lo traducen al lenguaje de la computadora.
💡

En la práctica: la próxima vez que uses una app cualquiera, trata de pensar en la lógica detrás. "Si el usuario no está conectado, muestra la pantalla de login. Si sí está conectado, muestra su perfil." Eso es todo — condiciones. Todo programa complejo es esa misma idea, apilada muchas veces.

Frontend y Backend

Las dos grandes mitades de cualquier aplicación

🍽️ Como un restaurante — el salón donde el cliente come y hace su pedido es el Frontend. La cocina donde se prepara la comida es el Backend. El cliente solo ve el salón, nunca la cocina. Pero sin cocina no hay comida.

Cualquier aplicación web tiene estas dos partes bien diferenciadas. Entender a qué se refiere cada una es fundamental para saber de qué hablan los tutoriales, los empleos y los desarrolladores cuando usan estos términos. 

FRONTEND                          BACKEND
────────────────                  ────────────────
Lo que el usuario VE              Lo que el usuario NO ve
y con lo que interactúa           pero hace que todo funcione

HTML → estructura                 Servidor → recibe peticiones
CSS  → diseño visual              Base de datos → guarda información
JS   → interactividad             Lógica → procesa y responde

Corre en el NAVEGADOR             Corre en un SERVIDOR
EJEMPLO 
// Cuando haces login en una app:

// FRONTEND (el formulario que ves)
// → El usuario escribe su correo y contraseña
// → JavaScript toma esos datos y los envía al servidor

// BACKEND (lo que pasa "detrás")
// → El servidor recibe el correo y contraseña
// → Los busca en la base de datos
// → Si coinciden, responde "acceso concedido"
// → Si no, responde "contraseña incorrecta"

// FRONTEND de nuevo
// → Recibe la respuesta del servidor
// → Muestra el mensaje al usuario o lo redirige
¿Qué aprenderás en esta guía?

Esta guía se enfoca en Frontend — HTML, CSS y JavaScript. Es el lugar ideal para empezar porque ves los resultados al instante en el navegador sin necesitar configurar servidores ni bases de datos.

El Backend se aprende después, cuando ya tienes sólidas las bases. Usa lenguajes como Node.js, Python, PHP o Java, y tecnologías como MySQL, PostgreSQL o MongoDB para las bases de datos.
💡

Término relacionado — Full Stack: Un desarrollador que sabe tanto Frontend como Backend se llama Full Stack Developer. Es el perfil más buscado en la industria. Empezando por el Frontend ya vas en la dirección correcta.

Cliente y Servidor

Quién pide y quién responde en internet

📞 Como una llamada telefónica — tú llamas (cliente) y alguien al otro lado contesta (servidor). Tú haces una petición, el servidor la procesa y te responde. Todo lo que pasa en internet es una variación de esta conversación.

Estos dos términos aparecen constantemente en programación y es fundamental saber a qué se refieren. No son hardware específico — son roles en una comunicación. 

CLIENTE                           SERVIDOR
────────────────                  ────────────────
Tu navegador (Chrome, Firefox)    Una computadora en internet
Tu app en el celular              que está siempre encendida
Tu programa de escritorio         esperando peticiones

Hace PETICIONES (requests)        Recibe peticiones
                                  Las procesa
Recibe RESPUESTAS (responses)     Devuelve respuestas

Ejemplo:                          Ejemplo:
Tú escribes                       El servidor de Google
"google.com" en el               recibe tu petición,
navegador → eso es               busca la página
una petición                      y te la envía

La comunicación entre cliente y servidor usa un protocolo llamado HTTP (HyperText Transfer Protocol). Cuando ves https:// al inicio de una dirección web, estás viendo ese protocolo. La s al final significa que la conexión es segura (cifrada).

FLUJO 
// Lo que pasa cuando abres una página web:

// 1. Escribes la URL en el navegador
"https://google.com"

// 2. El cliente (navegador) hace una petición al servidor
// "Hola servidor de Google, mándame tu página principal"

// 3. El servidor procesa la petición y responde
// "Aquí tienes: [el HTML de la página]"

// 4. El navegador recibe el HTML, CSS y JS
// y los convierte en la página visual que ves

// Todo esto ocurre en milisegundos
Términos que escucharás siempre:

Request (petición) — cuando el cliente pide algo al servidor
Response (respuesta) — cuando el servidor contesta
HTTP — el protocolo de comunicación
URL — la dirección de un recurso en internet
Status code — el código de respuesta del servidor (200 = éxito, 404 = no encontrado, 500 = error del servidor)

¿Qué es una API?

El enchufe que conecta tu app con el mundo

🔌 Como un enchufe en la pared — no necesitas saber cómo funciona la red eléctrica de la ciudad para enchufar tu cargador y cargar tu celular. La API es ese enchufe: una interfaz estándar que te permite usar el servicio de otro sin entender cómo funciona por dentro.

API significa Application Programming Interface — Interfaz de Programación de Aplicaciones. Es un conjunto de reglas que define cómo dos programas pueden comunicarse entre sí. Cuando tu app pide datos al clima, a Google Maps, a Twitter o a cualquier servicio externo, lo hace a través de una API. 

TU APP          API              SERVICIO EXTERNO
────────        ────────         ────────────────
"Dame el        Traduce tu       Base de datos
clima de        petición         del servicio
Córdoba"   →   al formato   →   meteorológico
               que el
               servicio        Busca Córdoba
               entiende        Encuentra datos
                               
               Trae la     ←   Devuelve datos
Tu app     ←   respuesta
muestra
"25°C ☀️"

Las APIs responden con datos en formato JSON — el mismo formato de objetos JavaScript que ya aprendiste. Por eso aprender objetos y arreglos primero fue tan importante: cuando consumas APIs, los datos llegan exactamente en ese formato.

EJEMPLO JSON 
// Así responde una API del clima, por ejemplo:
{
  "ciudad":      "Córdoba",
  "temperatura": 25,
  "condicion":   "soleado",
  "humedad":     62
}

// Tu app recibe ese objeto y lo muestra en pantalla
// ciudad → "Córdoba"
// temperatura → 25
// condicion → "soleado"
APIs que existen en el mundo real:

Google Maps API — mostrar mapas en tu app
OpenWeather API — datos del clima en tiempo real
Stripe API — procesar pagos
Firebase API — base de datos y autenticación
Cloudinary API — subir y gestionar imágenes
Twitter/X API — publicar tweets desde código

Hay miles de APIs públicas y gratuitas para aprender. Cuando llegues al capítulo de APIs en esta guía, aprenderás a consumir cualquiera de ellas.

La Terminal

La herramienta más poderosa de cualquier programador

🗺️ Como hablar directamente con la computadora sin intermediarios — en lugar de hacer clic en íconos y menús (interfaz gráfica), escribes comandos en texto y la computadora los ejecuta. Es más difícil al principio, pero infinitamente más poderoso. Todo lo que haces con clics en 10 segundos, en la terminal lo haces en 2.

La terminal (también llamada consola, línea de comandos, CLI o command prompt) es una ventana donde escribes instrucciones directamente al sistema operativo. Es la herramienta que usan todos los desarrolladores profesionales todos los días. En Windows se llama CMD o PowerShell. En Mac y Linux se llama Terminal.

COMANDOS BÁSICOS 
# Navegar entre carpetas
cd NombreCarpeta      ← entrar a una carpeta
cd ..                 ← subir un nivel (carpeta anterior)
cd ~                  ← ir a la carpeta principal del usuario

# Ver el contenido de una carpeta
ls                    ← Mac/Linux: listar archivos
dir                   ← Windows: listar archivos

# Crear carpetas y archivos

La terminal también es donde instalas herramientas, corres servidores, usas Git, instalas librerías y ejecutas código. Casi todo el trabajo profesional de desarrollo web pasa por ella en algún momento.

¿Por qué aprender la terminal si ya tengo interfaz gráfica?

Porque muchas herramientas de desarrollo solo existen en la terminal. Git, npm (el gestor de paquetes de JavaScript), Node.js y muchos otros no tienen interfaz de clic — solo línea de comandos.

Además, cuando algo falla en producción a las 3am, la terminal es lo único disponible para diagnosticar y arreglar el problema. Acostumbrarse a ella desde el principio marca la diferencia.
💡

Primer ejercicio: Abre la terminal de tu computadora ahora mismo. En Windows: busca "cmd" o "PowerShell" en el menú inicio. Escribe dir y presiona Enter. Verás la lista de archivos y carpetas donde estás. Eso ya es usar la terminal.

Git y GitHub

La máquina del tiempo de tu código

📸 Como tomar fotos de tu código en diferentes momentos — si algo se rompe, puedes volver a la foto de ayer cuando todo funcionaba. Git guarda esas fotos en tu computadora. GitHub las guarda también en internet para que nunca las pierdas y puedas trabajar desde cualquier lugar.

Git es un sistema de control de versiones — un programa que installas en tu computadora y que lleva registro de cada cambio que haces en tu código. Cada vez que guardas una "foto" (llamada commit) Git recuerda exactamente cómo estaba todo.

GitHub es un sitio web donde subes esas fotos a internet. Es como Google Drive pero para código. Es donde la mayoría de proyectos de software del mundo están almacenados — incluyendo el tuyo. 

TU COMPUTADORA                    INTERNET
────────────────                  ────────────────
Tu código                         GitHub
    ↓                                 ↑
Git (control local)   →  push  →  Repositorio remoto
    ↑                                 ↓
  commit              ←  pull  ←  (otros pueden verlo)
GIT BÁSICO 
# Los comandos más usados de Git (en la terminal)

git init              ← inicializa Git en una carpeta
git status            ← muestra qué archivos cambiaron
git add .             ← selecciona TODOS los cambios para guardar
git commit -m "mensaje"  ← toma la "foto" con una descripción
git push              ← sube los cambios a GitHub
git pull              ← baja cambios de GitHub a tu computadora
git log               ← ve el historial de todas las fotos
git checkout nombreFoto  ← regresa a una foto anterior
Vocabulario de Git que escucharás siempre:

Repositorio (repo) — la carpeta de tu proyecto con todo el historial de Git
Commit — una "foto" guardada del estado del código con un mensaje descriptivo
Branch (rama) — una versión paralela de tu código para probar cosas sin romper lo principal
Merge — unir dos ramas en una
Clone — descargar un repositorio de GitHub a tu computadora
Fork — copiar el repositorio de otra persona para modificarlo
Pull Request — proponer que tus cambios se integren al proyecto principal
💡

Ejemplo en la práctica: Si estás trabajando en tu guía y de repente algo se rompe y no sabes qué fue, con Git haces git log para ver el historial, eliges la última versión que funcionaba y la restauras en segundos. Sin Git tendrías que recordar manualmente qué cambiaste o reescribirlo todo.

Bases de datos

Donde viven los datos de forma permanente

🗄️ Como un archivero físico gigante con cajones organizados — cada cajón guarda un tipo de información: usuarios, productos, mensajes. Puedes buscar, agregar, modificar y borrar información en cualquier momento. Sin base de datos, toda la información desaparece cuando cierras el navegador.

Las variables y arreglos de JavaScript guardan datos solo mientras la página está abierta. Cuando cierras el navegador, desaparecen. Las bases de datos guardan información de forma permanente en un servidor, independientemente de quién esté conectado o no.

EJEMPLO SQL 
-- SQL: el lenguaje para hablar con bases de datos relacionales

-- Crear una tabla de usuarios
CREATE TABLE usuarios (
  id      INTEGER,
  nombre  TEXT,
  correo  TEXT,
  edad    INTEGER
);

-- Insertar un usuario
INSERT INTO usuarios VALUES (1, "Ana", "ana@mail.com", 22);

-- Buscar todos los usuarios
SELECT * FROM usuarios;

-- Buscar usuarios mayores de 18
SELECT nombre FROM usuarios WHERE edad >= 18;
¿Cuándo necesitas una base de datos?

Cuando la información necesita persistir (sobrevivir al cierre del navegador), cuando múltiples usuarios necesitan acceder a los mismos datos, o cuando la cantidad de datos es demasiado grande para guardarla en el código.

Ejemplos: usuarios registrados, historial de compras, mensajes de un chat, publicaciones de una red social, progreso en un videojuego.
💡

Ejemplo en la práctica: Cuando inicias sesión en cualquier app, el servidor consulta la base de datos con tu correo, encuentra tu cuenta, verifica tu contraseña y te da acceso. Sin base de datos no habría forma de recordar que eres tú entre una visita y otra.

Frameworks y librerías

Herramientas que otros construyeron para que no empieces de cero

🏠 Como comprar una casa prefabricada en lugar de construirla ladrillo por ladrillo — alguien ya resolvió los problemas más comunes y te da la estructura lista. Tú solo personalizas lo que necesitas. Una librería es como comprar una puerta ya hecha. Un framework es como comprar la estructura completa de la casa.

Cuando aprendes JavaScript desde cero, escribes todo a mano. En el trabajo real, nadie escribe todo desde cero — usan herramientas que otros ya construyeron. Entender qué son estas herramientas es importante para saber de qué hablan cuando mencionan React, Vue, Angular, etc.

cómo usarla.                     estructura — tú llevas el
                                  contenido.

Ejemplos:                         Ejemplos:
Chart.js → gráficas               React → interfaces de usuario
Lodash → utilidades JS            Vue → interfaces reactivas
Axios → peticiones HTTP           Angular → apps empresariales
GSAP → animaciones                Next.js → apps con React
                                  Express → backend con Node.js
¿Cuándo aprender frameworks?

Después de dominar las bases. Aprender React sin entender JavaScript es como aprender a manejar en autopista sin haber aprendido en ciudad. Técnicamente funciona, pero cuando algo falla no entiendes por qué.

El orden correcto es:
HTML + CSS + JavaScript puro → Git + Terminal → un Framework → APIs → Backend

Esta guía te lleva exactamente por ese camino.

Glosario esencial

Los términos que todo desarrollador usa a diario

🗣️ Como aprender el idioma antes de mudarte a otro país — no necesitas hablar perfecto desde el día uno, pero sí entender las palabras que más se repiten. Este glosario es ese vocabulario básico que te permite entender a otros desarrolladores, tutoriales y documentación.

Estos son los términos que aparecen constantemente en cualquier tutorial, curso, video o conversación sobre programación web. Con este vocabulario claro, el resto del aprendizaje va mucho más rápido.
